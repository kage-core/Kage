kage://docs

Documentation for repo memory, code graphs, and agent recall.

Kage is a CLI and MCP server that gives coding agents repo-local memory. It stores reusable learnings as reviewable packets, connects them to code, and recalls the right context before future edits.

Install

Install the npm package globally, then initialize a repo. Kage writes repo-local memory under .agent_memory/ and installs the agent policy in AGENTS.md.

npm install -g @kage-core/kage-graph-mcp
kage init --project .
kage doctor --project .
Repo-local by default.

Kage does not need a hosted database for normal use. Memory packets are plain JSON and generated graph artifacts stay in the repo memory directory.

Agent setup

Use setup commands to add Kage MCP configuration for your agent. Restart the agent after setup so the MCP server and repo policy are loaded.

Codex

kage setup codex --project . --write
kage setup verify-agent --agent codex --project .

Claude Code

kage setup claude-code --project . --write
kage setup verify-agent --agent claude-code --project .

Other supported setup targets include Cursor, Windsurf, Gemini CLI, OpenCode, Cline, Goose, Roo Code, Kilo Code, Claude Desktop, Aider, and generic MCP clients.

Daily workflow

Before work

Agents call kage_context or the CLI recall path to load relevant memory and code graph facts for the task.

kage recall "how does auth work?" --project . --explain

After learning

Capture durable repo knowledge: decisions, gotchas, setup commands, test workflows, and code explanations.

kage learn --project . --type gotcha --learning "Do not merge these retry paths; one uses idempotency keys and the other uses session state."

After meaningful changes

Refresh generated artifacts so recall, graph views, metrics, and stale checks match the current tree.

kage refresh --project .
kage pr check --project .

Optional automation

Install a repo-local post-commit hook when you want Kage to refresh memory and branch summaries after commits without replacing existing hook content.

kage hook install --project .
kage hook status --project .

Before handoff

Create branch memory from the diff and run the merge-readiness check.

kage propose --project . --from-diff
kage pr check --project . --json

CLI reference

The CLI is the user-facing control surface. Use --json where automation needs structured output.

kage init Initialize a repo

Creates repo memory structure, installs policy, and builds the first index.

kage init --project .

kage setup Configure agents

Lists setup targets, writes agent config, and verifies live agent integration.

kage setup list
kage setup codex --project . --write
kage setup verify-agent --agent codex --project . --json

kage recall Retrieve relevant memory

Searches repo-local packets and returns an agent-ready context block. Use --explain to see why packets matched.

kage recall "query" --project . --json --explain

kage learn Capture a learning

Writes a concise repo-local memory packet for a reusable fact learned during work.

kage learn --project . --type decision --title "Parser tests need fixtures" --learning "Run parser fixture tests after grammar edits." --paths mcp/kernel.ts,mcp/kernel.test.ts

kage capture Create a structured packet

Writes a packet with explicit title, summary, body, type, tags, paths, and stack metadata.

kage capture --project . --title "Auth retry gotcha" --body "..." --type gotcha --tags auth,retry

kage refresh Rebuild generated intelligence

Updates indexes, structural artifacts, code graph, memory graph, metrics, and stale-memory checks. Use --full only when you need to bypass cache.

kage refresh --project .
kage refresh --project . --full --json

kage pr Prepare a branch

Summarizes branch memory and verifies validation, stale packets, graph freshness, and memory changes.

kage pr summarize --project .
kage pr check --project . --json

kage hook Automate post-commit refresh

Installs, checks, or removes a marker-delimited git post-commit hook. The hook preserves existing hook content and runs kage refresh plus kage pr summarize after commits. Set KAGE_SKIP_HOOK=1 to bypass it.

kage hook install --project .
kage hook status --project . --json
kage hook uninstall --project .

kage viewer Inspect memory, graphs, and reports

Starts the local viewer with repo graph, code graph, metrics, review queue, pending memory, and repo-intelligence reports preloaded.

kage viewer --project . --port 3113

kage graph Query memory graph

Queries evidence-backed entities and relationships generated from approved memory packets.

kage graph --project . --json
kage graph "checkout retry" --project . --json
kage graph --project . --mermaid

kage code-graph Query source graph

Queries files, symbols, imports, calls, routes, tests, and package scripts generated from source. For non-TypeScript code, the generic indexer also extracts bounded call edges, test function coverage signals, and Python framework routes for FastAPI, Flask, and Django; SCIP, LSP, LSIF, and tree-sitter artifacts override generic facts when present.

kage code-graph --project . --json
kage code-graph "recall ranking" --project . --json

kage dependency-path Explain file connections

Finds the shortest code graph path between two files and reports whether the source depends on the target, the target depends on the source, or the connection is only undirected.

kage dependency-path --project . --from src/app.ts --to src/auth.ts --json

kage cleanup-candidates Find conservative cleanup candidates

Reports unreferenced source-file candidates with confidence and reasons. It does not delete code; use it as review input before cleanup PRs.

kage cleanup-candidates --project . --json

kage reviewers Suggest local reviewers

Ranks reviewers for target or changed files using local git authorship, recent edits, and co-change ownership. It does not call GitHub or external services.

kage reviewers --project . --changed-files src/auth.ts,src/session.ts --json

kage contributors Profile local ownership

Builds contributor profiles from local git history: commits, recent activity, touched files and modules, primary-owned files, ownership silos, hotspot ownership, and commit category mix.

kage contributors --project . --json

kage decisions Audit why-memory coverage

Reports decisions, gotchas, runbooks, conventions, constraints, code explanations, weak or stale memories, and important code paths that still lack linked decision memory.

kage decisions --project . --json

kage module-health Score local modules

Rolls up code graph, test, cleanup, churn, and ownership signals into local module scorecards for review planning.

kage module-health --project . --json

kage graph-insights Summarize architecture signals

Reports language parser coverage, edge mix, central files, dependency cycles, import communities, and short entry flows from the source graph.

kage graph-insights --project . --json

kage workspace Recall across sibling repos

Scans a local workspace for git repos, reports Kage memory coverage, detects package dependencies and route contract links between repos, and can fan out recall across every indexed repo.

kage workspace --project .. --json
kage workspace recall "auth header contract" --project .. --json

kage risk Check change risk before editing

Combines the code graph with local git history to report dependents, impact surface, churn, ownership, ownership silos, co-change partners missing from the change set, hotspots, and missing test signals for target files.

kage risk --project . --targets src/auth.ts --json
kage risk --project . --changed-files src/auth.ts,src/session.ts

kage code-index Build source indexes

Writes external code index artifacts. It prefers SCIP where available and falls back to built-in indexing.

kage code-index --project . --json
kage structural-index --project . --json

kage metrics Show adoption and graph metrics

Reports code graph coverage, memory graph evidence coverage, useful memory ratio, validation state, and readiness.

kage metrics --project . --json

kage benchmark Estimate recall impact

Reports coverage and compares baseline discovery cost against Kage context for a task.

kage benchmark --project . --json
kage benchmark --project . --compare --task "fix flaky checkout retry" --json

kage inbox Review memory work

Shows pending packets, stale packets, duplicates, validation issues, and recommended actions.

kage inbox --project . --json
kage review-artifact --project .

kage audit Check trustworthiness

Audits memory health, structured context coverage, code graph quality, graph links, and concrete recommendations.

kage audit --project . --json
kage quality --project . --json
kage validate --project .

kage org/global Explicit sharing flows

Org, global, public, registry, and marketplace surfaces are review-gated. They do not publish automatically.

kage org status --project . --org acme --json
kage org upload --project . --org acme --packet <approved-packet-id>
kage org review --project . --org acme --packet <org-packet-id> --approve
kage layered-recall "query" --project . --org acme --global --json
kage promote --project . --public <packet-id>
kage export-public --project .
kage global build --project . --json

MCP tools

Agents should prefer MCP tools when available. The most important tool is kage_context, which combines validation, recall, code graph search, knowledge graph search, and target-aware risk signals.

kage_context Main task entry point

Call before repo-specific implementation, debugging, architecture, or code questions. When a query or target list includes file paths, Kage also adds risk and dependency-path context.

kage_recall, kage_graph, kage_code_graph

Lower-level recall and graph tools when the combined context tool is not enough.

kage_risk Pre-edit risk assessment

Use before changing shared or high-churn files. Returns graph dependents, git churn, ownership, ownership silos, co-change warnings, impact surface, and test-gap signals.

kage_dependency_path Code graph path finder

Use when an agent needs to understand how two files connect before changing a flow or explaining impact.

kage_cleanup_candidates Cleanup review input

Returns conservative unreferenced-file candidates from the code graph with confidence and reasons. It never mutates files.

kage_reviewers Local reviewer suggestions

Suggests reviewers from git history and co-change ownership for target or changed files.

kage_contributors Contributor profiles

Returns local git profiles with commits, recent activity, touched files/modules, primary ownership, ownership silos, hotspot ownership, and commit category mix.

kage_decisions Why-memory coverage

Returns decision, gotcha, runbook, convention, constraint, and code explanation coverage by path, plus important uncovered code paths and weak or stale memories.

kage_module_health Module health scorecards

Returns local module scorecards from code graph, tests, cleanup candidates, churn, and ownership concentration.

kage_graph_insights Graph architecture signals

Returns language parser coverage, edge mix, central files, dependency cycles, import communities, and entry flows for broad architecture orientation.

kage_workspace, kage_workspace_recall

Summarizes sibling repos, package dependencies, route contract links, and recalls Kage memory across a local multi-repo workspace without copying packets or requiring a hosted database.

kage_learn, kage_capture, kage_feedback

Capture new repo knowledge, write structured packets, or mark recalled memory helpful, wrong, or stale.

kage_refresh, kage_pr_summarize, kage_pr_check

Keep generated artifacts current and create merge-ready branch memory.

kage_metrics, kage_audit, kage_inbox, kage_quality, kage_benchmark

Inspect readiness, quality, benchmark estimates, duplicates, stale packets, and review actions.

kage_setup_agent, kage_verify_agent, kage_install_policy

Install or verify agent harness configuration.

kage_search, kage_fetch, kage_list_domains

Optional community knowledge graph search. Treat returned public context as advisory and untrusted compared with repo memory.

kage_org_status, kage_org_upload_candidate, kage_org_recall, kage_layered_recall, kage_global_build

Review-gated sharing and layered recall. Repo-local memory wins when context conflicts.

Memory model

Kage stores reusable learnings as packets, not raw chat transcripts. A good packet is short, source-backed, path-grounded, and useful to a future agent.

  • runbook: how to run, test, build, or debug.
  • bug_fix: cause, fix, evidence, and verification.
  • decision: what was decided and why.
  • rationale: why a tradeoff exists.
  • convention: a repo rule agents should follow.
  • workflow: repeated steps for a task.
  • gotcha: a failure mode to avoid.
  • code_explanation: why non-obvious code exists.
  • negative_result: what was tried and did not work.
  • constraint: a limitation that changes implementation choices.
repo-local reviewable JSON path-grounded source-backed not transcripts

Graphs and viewer

Kage maintains two related graph surfaces: a memory graph generated from packets and a source-derived code graph generated from code. The viewer overlays them and adds a Repo Intelligence cockpit so humans can inspect what was recalled, why it matters, and what risk or graph signals are attached to the repo.

Memory graph

Entities, episodes, and evidence-backed edges from approved packets.

Code graph

Files, symbols, imports, calls, tests, routes, and source relationships.

kage viewer --project .
kage graph "retry logic" --project . --json
kage code-graph "checkout tests" --project . --json

The local viewer writes and loads .agent_memory/reports/*.json for risk, contributors, decisions, module health, graph insights, workspace coverage, quality, and benchmark proof. The hosted viewer at viewer/ is useful for static demos. The local viewer is preferred for real repos because it loads the active repo artifacts automatically.

Review and sharing

Kage is collaborative because repo memory travels with the repository and can be reviewed like code. Sharing outside the repo is explicit.

  • Repo packets live under .agent_memory/packets/.
  • Generated indexes and graph artifacts can be refreshed from the repo.
  • Branch change memory is created from diffs with kage propose --from-diff.
  • PR readiness is checked with kage pr check.
  • Org/global/public promotion requires human review commands.

Large repos

Kage uses cache-backed structural indexing and generated artifacts so repeat refreshes can reuse unchanged work. Full rebuilds should be deliberate.

  • Run normal kage refresh for day-to-day updates.
  • Use kage hook install when the repo should refresh generated Kage state after commits.
  • Use kage refresh --full when changing indexer behavior or recovering from stale artifacts.
  • Use kage metrics to inspect indexable files, ignored files, cache hits, and parser coverage.
  • Use kage gc to inspect or clean generated artifacts when needed.

Troubleshooting

Agent does not call Kage

Verify setup and restart the agent so the MCP server and repo policy are loaded.

kage setup verify-agent --agent codex --project .
kage doctor --project .

Recall returns stale or wrong memory

Mark the packet as stale or wrong, then refresh and review the inbox.

kage feedback --project . --packet <packet-id> --kind stale
kage inbox --project . --json

Viewer has no graph

Launch it from the repo with kage viewer, or open the hosted viewer with graph/code/metrics URL parameters.

kage viewer --project .

Refresh seems slow

Check cache hits, indexable file counts, ignored files, and parser coverage before forcing a full rebuild.

kage metrics --project . --json
kage audit --project . --json
