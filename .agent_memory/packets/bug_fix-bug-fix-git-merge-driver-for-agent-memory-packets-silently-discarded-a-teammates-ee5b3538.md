# BUG FIX: git merge driver for .agent_memory packets silently discarded a teammate's concurrent edit

> mergePacketFiles mcp/kernel.ts is last write wins by each side's self reported updated at, not a field level three wa…

mergePacketFiles() (mcp/kernel.ts) is last-write-wins by each side's self-reported updated_at, not a field-level three-way merge (the base arg was explicitly unused: 'Reserved for a future field-level three-way merge'). When two teammates concurrently edited the SAME packet file (both reverified it, or one approved while another superseded it), the losing side vanished from disk with no trace on merge. This mirrors a pattern already solved for PERSONAL cross-machine sync (kage sync preserves losers under ~/.kage/memory/conflicts/, per docs/CLOUD.md) but was missing for the repo-team merge path. FIX: mergePacketFiles() now takes an optional projectDir; when both sides parse and genuinely diverge (not a no-op race where content is identical), the losing side is written to .agent_memory/conflicts/<id>-lost-<timestamp>.md via the new conflictsDir() helper, and the merge driver's stderr message (what git shows during a real merge) reports the preserved path. Best-effort only: a failure to preserve never blocks the merge. cli.ts's merge-packet command now passes process.cwd() as projectDir (the real repo root during an actual git-invoked merge).
Evidence: Test 'merge-packet never silently drops a teammate's concurrent edit' in mcp/kernel.test.ts reverse-engineers the exact race (two divergent edits to one packet, base included) and asserts the loser's real body content is recoverable from the preserved file afterward; also asserts a true no-op race (identical resolution) does not spam a conflict artifact, and that omitting projectDir keeps the fix best-effort/non-blocking.
