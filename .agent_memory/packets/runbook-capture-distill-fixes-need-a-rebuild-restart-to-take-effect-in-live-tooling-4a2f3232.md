# Capture/distill fixes need a rebuild+restart to take effect in live tooling

> A code fix to Kage's capture / distill / recall path does NOT take effect in the running session's live tooling until…

A code fix to Kage's capture / distill / recall path does NOT take effect in the running session's live tooling until the global kage binary is rebuilt/reinstalled AND the agent is restarted. Two reasons: (1) the Stop/observe hooks at ~/.claude/kage/hooks/*.sh shell out to the `kage` command on PATH (`kage distill --auto`, `kage refresh`, `kage pr summarize`, `kage reconcile`), which is the globally installed published build; (2) the MCP server (kage_context etc.) loads its dist at session start. Symptom observed: after fixing the dump-capture guard and deleting 7 dump packets, the Stop hook's `kage distill --auto` (old code, no guard) re-created dump packets mid-session from this session's own observations — including one that captured a subagent's /tmp slice-file reads. To verify a capture/recall fix, run the freshly-built CLI at mcp/dist/cli.js directly (e.g. `node mcp/dist/cli.js recall ...`), NOT the live MCP tools or hooks. To make the fix live: rebuild (npm run build in mcp/), reinstall/relink the global binary, then restart the agent so the MCP server reloads.
Evidence: Stop hook stop.sh lines 10-37 invoke PATH `kage`; dump packets rec-e39ccba3 and funnel-diagnosis 7dae01b9 reappeared after deletion during this session; verified the fix via node mcp/dist/cli.js (270063->6009 chars) while the live MCP tool still used old code.
Verified by: observed dump regeneration via Stop hook + measured fix via built dist CLI
