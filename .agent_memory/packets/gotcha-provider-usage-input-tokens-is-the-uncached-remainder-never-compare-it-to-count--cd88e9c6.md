# Provider usage.input_tokens is the UNCACHED remainder — never compare it to count_tokens

> In the Anthropic API, response usage.input tokens is the UNCACHED REMAINDER ONLY. The full prompt size is input token…

In the Anthropic API, response `usage.input_tokens` is the UNCACHED REMAINDER ONLY. The full prompt size is input_tokens + cache_creation_input_tokens + cache_read_input_tokens. The /v1/messages/count_tokens endpoint, by contrast, returns the FULL token count of the body and knows nothing about caching.

These two numbers are therefore NOT commensurable, and Kage's vNext proxy receipts originally put one on each side of an "exact" measurement. Claude Code — the proxy's actual user — uses cache_control aggressively, so in steady state usage.input_tokens is a few hundred against a prompt of tens of thousands. The receipt would read: before 26,300 tokens / $0.13, after 420 tokens / $0.002, measurement_quality "exact" — i.e. Kage claiming that injecting context SAVED 98% of the prompt cost. In audit mode the same mechanism inverts and claims injection increased cost ~60x. That is an inference dressed as a measurement, and it is the exact failure the whole measurement phase exists to prevent.

Rules that follow, all now enforced in mcp/vnext/measurement/:
- The token quantity that goes on a receipt is totalPromptTokens = input + cache_creation + cache_read, which IS commensurable with count_tokens. If ANY component is absent or unparseable the total is null — never a partial sum, never defaulted to 0.
- Cost must be priced per component, because cached and uncached tokens bill at different rates (cache read ~0.1x, 5m cache write ~1.25x, 1h ~2x). Pricing a cached request as if all input were uncached is wrong. A side with a measured token count but NO measured cache breakdown (which is every count_tokens side) gets a NULL cost, not a cost computed as if uncached.
- Consequence for any aggregate: exact TOKEN deltas are available far more often than exact COST deltas, and a one-sided cost is UNUSABLE, not zero. Never conflate the two.
- /v1/messages/count_tokens rejects unrecognized top-level fields. Posting a full Messages body (with max_tokens, stream, metadata) 400s, so the probe silently returns null and every receipt stays "partial" forever while still costing a billable round trip. Send only model/messages/system/tools/tool_choice/thinking.
- Receipts are written only for requests that were ACTUALLY TRANSFORMED. A zero-recall request writes nothing, so exact/partial/unavailable coverage means "of transformed requests" — otherwise trivially-exact no-op rows drown the one number that matters.

Testing lesson: the original tests all used a fake upstream returning usage WITHOUT cache fields — the single regime where the two quantities coincide. A measurement test that never exercises a cached request cannot catch this class of bug.
Evidence: Found by review of commit 73c7a97 and fixed in 4fb4390. Every pre-existing test's fake upstream returned usage with no cache fields — the one regime where the two numbers happen to agree — which is exactly why the defect survived the first implementation and its tests.
