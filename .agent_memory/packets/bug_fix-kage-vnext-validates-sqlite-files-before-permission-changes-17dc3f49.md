# Kage vNext validates SQLite files before permission changes

> Kage vNext treats a file backed SQLite database and its wal/ shm sidecars as one filesystem trust set. It atomically …

Kage vNext treats a file-backed SQLite database and its -wal/-shm sidecars as one filesystem trust set. It atomically precreates an absent database as 0600, then lstat-classifies every existing DB/WAL/SHM path before changing any member's mode or constructing SQLite. Only regular non-symlink files are accepted; directories, symlinks, FIFOs, devices, sockets, and other special objects receive contextual Kage vNext path errors without following or mutating their targets. Valid legacy DB/WAL/SHM files are tightened to 0600 before open, and sidecars are classified and tightened again after WAL activation inside the close-on-initialization-failure boundary. Existing parent directory modes remain unchanged.
Evidence: The initial required regressions failed 4 of 36 tests: legacy WAL remained 0644, a directory was chmodded, a symlink target was chmodded, and SQLite opened despite a sidecar symlink. A two-phase ordering regression then failed 1 of 36 until all paths were classified before any chmod. Final focused storage tests passed 36/36; npm test --prefix mcp passed 470/470 plus 12/12 dogfood tests.
Verified by: npm run build --prefix mcp; node --test mcp/dist/vnext/storage/storage.test.js; npm test --prefix mcp
