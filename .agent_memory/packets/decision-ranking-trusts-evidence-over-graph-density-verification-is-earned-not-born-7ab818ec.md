> Two core fixes landed 2026 07 03 items 1 2 of the deep dive plan . RANKING recallBreakdown, mcp/kernel.ts : the knowl…

Two core fixes landed 2026-07-03 (items 1-2 of the deep-dive plan). RANKING (recallBreakdown, mcp/kernel.ts): the knowledge-graph prior is now log1p-damped and capped at parity with lexical evidence (min(textScore, 8), was textScore*1.5+12) so edge-dense old packets cannot outrank the packet whose title matches the query; recallRecencyScore adds +3 (<14d) → +1 (<60d) → 0 → -3 (>120d, changelog-shaped types only: decision/bug_fix/workflow/reference/issue_context — evergreen types never take the penalty); terse identifier queries (camelCase/snake_case tokens) ground through the code graph — a packet citing the file that defines the queried symbol gets +6 identifier score and survives the relevance>0 gate with zero lexical overlap. Verified live: "how do I run the tests" returns the test runbook #1; "observationSignalScore" surfaces the distill-gating memory. VERIFICATION HONESTY: packetVerificationLabel exports the rule — "verified" requires an actual recheck (freshness.verification !== "repo_local_agent_capture"); packets are born unverified in OKF export (okfVerifiedStatus). reverifyMemory now refuses a bare re-stamp when stored fingerprints changed (that path laundered false claims back to valid — proven by the deep dive); `kage reverify --evidence "..."` or `--verified-by "..."` is required for changed paths and records a reverification source_ref with prior→new shas; clean fingerprints refresh freely but do NOT upgrade verification. verifyCitations.ok is a real verdict now (fails on hard-stale or ungrounded; was hardcoded true). x-kage-confidence (hardcoded 0.7, never computed) removed from OKF frontmatter; recall/file-context displays show the verification label instead of "confidence 0.70".
