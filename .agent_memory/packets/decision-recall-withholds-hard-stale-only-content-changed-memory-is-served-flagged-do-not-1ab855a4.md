# Recall withholds hard-stale only; content-changed memory is served flagged (do not claim "wrong knowledge never reaches the agent")

> The recall path recallWithVectorScores in mcp/kernel.ts, ~line 10108, via recallHardStaleReason WITHHOLDS only hard s…

The recall path (recallWithVectorScores in mcp/kernel.ts, ~line 10108, via recallHardStaleReason) WITHHOLDS only hard-stale memory: packets whose cited files are ALL deleted, past TTL, status deprecated/superseded, or human/agent reported-stale. Suppression is recorded (kageSuppressedMemory), not silent. Hallucinated citations are separately rejected at WRITE time so they are never stored. BUT a packet whose cited file still exists with CHANGED CONTENT (refactored / behavior changed) is NOT hard-stale: it is still served to the agent, only FLAGGED via staleMemoryReasons ("linked path changed since memory was verified"). This is the open task #39 (strict recall: withhold soft-stale too). The staleness benchmark (benchmarks/staleness-kage.mjs) measures Kage serving content-changed memory ~50% of the time. CONSEQUENCE FOR MARKETING: the claim "your diff changes what it cites, wrong knowledge never reaches the agent" is an OVERCLAIM and was removed from the site on 2026-06-14. Honest phrasing: "deleted and stale citations get caught." Only restore the strong "never reaches" wording after task #39 ships (recall withholds content-changed memory, not just flags it).
