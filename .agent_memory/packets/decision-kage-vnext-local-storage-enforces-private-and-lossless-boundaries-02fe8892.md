# Kage vNext local storage enforces private and lossless boundaries

> Kage vNext local persistence is a strict trust boundary. openVnextDatabase retains the canonical Node 22.5+ lazy node…

Kage vNext local persistence is a strict trust boundary. openVnextDatabase retains the canonical Node 22.5+ lazy node:sqlite gate, makes only the immediate file-backed runtime directory private at 0700, sets the database to 0600 before WAL so DB/WAL/SHM remain private, and closes opened handles if initialization fails. Migration 001 validates the migration ledger, rejects every future version using BigInt-safe reads, applies non-conditional DDL transactionally, and uses CHECK constraints for safe nonnegative numeric ranges. Receipt writes validate counts, costs, and latency before SQLite. Event payloads and receipt transformations pass a shared lossless JSON validator before serialization and after parsing; unsupported/lossy inputs are rejected and stored corruption errors name the table, JSON column, and row identity.
Evidence: Four independent TDD cycles reproduced permissions/resource leaks, incompatible schema acceptance, invalid numeric persistence, and lossy/uncontextual JSON handling. Focused storage tests passed 25/25; npm test --prefix mcp passed 459/459 plus 12/12 dogfood.
Verified by: Task 2 quality-review red/green cycles and full MCP suite
