# Sessions auto-attach to the proxy via settings.local.json env + SessionStart ensure-up; hook code must survive set -euo pipefail and stale PATH kage

> Product decision Kushal, 2026 07 22 : 'kage run claude' is not acceptable UX — sessions must attach to the proxy auto…

Product decision (Kushal, 2026-07-22): 'kage run -- claude' is not acceptable UX — sessions must attach to the proxy automatically. Implementation: kage setup claude-code --write (and kage install) merges env.ANTHROPIC_BASE_URL=http://localhost:8788 into the PROJECT'S .claude/settings.local.json (personal, gitignored); upsertJsonSettings gained an env deep-merge where the USER'S existing values always win (a corporate gateway BASE_URL is never clobbered; removing the key detaches). The dead-proxy footgun is handled by a guarded ensure-up block in session-start.sh: if the repo is attach-wired (grep for ANTHROPIC_BASE_URL in settings.local.json) and nothing listens on 8788, spawn kage up with the mode recorded in .agent_memory/daemon/proxy.json (audit default; proxy.json survives reboots but is deleted by kage down). Three hook-authoring gotchas proven by real failures during end-to-end verification: (1) resolver order matters — a stale GLOBAL kage on PATH (published 3.3.0, predates 'kage up') silently no-ops; resolve repo-local CLI first ($CWD/node_modules/@kage-core/kage-graph-mcp/dist/cli.js, then $CWD/mcp/dist/cli.js for this dev repo), then PATH, then npx; (2) under the hooks' set -euo pipefail, an ASSIGNMENT from a failing command substitution is FATAL (VAR="$(sed ... missing-file | head -1)" kills the script before any || default applies) — put || true INSIDE the substitution; (3) verification of self-healing must be hook-ONLY — an earlier 'proof' was confounded by a manual kage up racing the hook; the definitive cycle is: down -> pipe {cwd} into session-start.sh -> poll port (healed in ~2s, exit 0, JSON output intact). plugin/hooks are generated from setupAgent — run kage gen-plugin-hooks after any hook-script change or the drift-guard test fails.
Evidence: Definitive cycle: kage down -> session-start.sh alone -> port 8788 listening in ~2s, hook exit 0, JSON intact. Earlier failures reproduced: PATH kage (3.3.0) no-op; hook exit 1 at the sed assignment under set -e until || true moved inside the substitution. Suite 1478/1478 including 4 new auto-attach tests + the plugin-hooks drift guard.
Verified by: npm test --prefix mcp; hook-only self-heal cycle; node --test mcp/dist/auto-attach.test.js

## Verification

(1) resolver order matters — a stale GLOBAL kage on PATH (published 3.3.0, predates 'kage up') silently no-ops; resolve repo-local CLI first ($CWD/node_modules/@kage-core/kage-graph-mcp/dist/cli.js, then $CWD/mcp/dist/cli.js for this dev repo), then PATH, then npx; (2) under the hooks' set -euo pipefail, an ASSIGNMENT from a failing command substitution is FATAL (VAR="$(sed ... missing-file | head -1)" kills the script before any || default applies) — put || true INSIDE the substitution; (3) verification of self-healing must be hook-ONLY — an earlier 'proof' was confounded by a manual kage up racing the hook; the definitive cycle is: down -> pipe {cwd} into session-start.sh -> poll port (healed in ~2s, exit 0, JSON output intact). plugin/hooks are generated from setupAgent — run kage gen-plugin-hooks after any hook-script change or the drift-guard test fails.
