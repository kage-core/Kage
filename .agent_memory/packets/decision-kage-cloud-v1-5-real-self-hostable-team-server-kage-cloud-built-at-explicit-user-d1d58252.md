# Kage Cloud v1.5: real self-hostable team server (kage cloud) built at explicit user request, ahead of the v2 SaaS waitlist gate

> Built a real, tested, self hostable team memory server mcp/cloud server.ts, mcp/cloud client.ts implementing docs/CLO…

Built a real, tested, self-hostable team memory server (mcp/cloud-server.ts, mcp/cloud-client.ts) implementing docs/CLOUD.md's v2 architecture as a real self-hosted v1.5, not a managed SaaS (the managed-SaaS waitlist gate on issue #74 is UNCHANGED and NOT satisfied by this — logged explicitly in docs/CLOUD.md's decision log). Zero new dependencies: node:http + node:sqlite (DatabaseSync). Real bearer-token auth (sha256-hashed, never stored plaintext), team namespaces, and a REAL review gate: a submitter's own token cannot approve their own packet (403 self_approval_blocked) — enforced server-side, not cosmetic. Client integration: kage cloud push/pull/list/approve/reject/invite/create-team/serve. Pulled packets land in a new teamPacketsDir (.agent_memory/team/packets, NOT the repo-authored packetsDir — kept separate so gc/refresh/the merge driver's repo-packet assumptions never touch server-sourced state) and surface in recall's new '## Team Memory' section via teamRecallEntries(), which re-verifies every packet's citations against the LOCAL checkout via the same recallStaleReason() repo/personal memory already use — 'verification stays client-side' as originally designed, now proven over a real network hop: a packet the server considers approved is still correctly withheld locally if the citation can't be verified on that machine. Proven live end-to-end via real CLI subprocesses (not just tests): kage cloud serve, create-team, push, self-approval blocked, real second-teammate approval, pull into a DIFFERENT repo, surfaces correctly in recall with author attribution.
Evidence: 3 real integration tests in mcp/cloud-server.test.ts (full team lifecycle incl. self-approval block + real approval + cross-repo pull + recall surfacing; auth rejection; stale-pull-withheld-locally). Live CLI proof: real kage cloud serve process, real create-team/push/invite/approve/pull round trip across two different demo repos, packet appeared in the second repo's real recall output labeled [team]. 399/399 full suite pass.
Verified by: npm test: 399/399 pass; live kage cloud serve + CLI round trip
