# Kage vNext storage preserves host ownership and verifies lossless schema claims

> Kage vNext file backed storage must preserve the mode and ownership policy of any pre existing parent directory. It c…

Kage vNext file-backed storage must preserve the mode and ownership policy of any pre-existing parent directory. It creates only a missing immediate runtime directory as 0700, atomically precreates a missing database as 0600 before SQLite opens it, tightens existing database files to 0600, and keeps WAL/SHM inheritance and initialization cleanup private. Lossless JSON and receipt numeric boundaries reject negative zero because JSON and SQLite normalize it to zero. A schema_migrations version-1 row is a claim, not proof: migration acceptance validates every required table's exact ordered columns, declared types, nullability, and primary-key positions plus unique single-column indexes for evidence_events.source_fingerprint and transformation_receipts.request_id. Fresh migration validates before recording version 1, and incompatible existing schemas roll back without repair.
Evidence: Three isolated red-green cycles reproduced the issues at 25/27, 27/29, and 29/32. The final focused suite passed 32/32; npm test --prefix mcp passed 466/466 plus 12/12 dogfood tests.
Verified by: npm run build --prefix mcp; node --test mcp/dist/vnext/storage/storage.test.js; npm test --prefix mcp
