# Kage Cloud dashboard shows the FULL packet — body, cited paths, tags — not a truncated summary, so approve/reject is an informed trust decision

> Fixed a real usability gap in GET /dashboard mcp/cloud server.ts : packet cards originally showed only title + a 160 …

Fixed a real usability gap in GET /dashboard (mcp/cloud-server.ts): packet cards originally showed only title + a 160-char truncated summary/body slice, which is not enough information to responsibly approve or reject a memory claim into team-shared trust. Cards now render: the full packet body (escaped, pre-wrapped, unclamped), the packet's cited paths as inline code chips (or an explicit orange 'No cited paths — cannot be re-verified against a checkout's code' warning when there are none, since an ungrounded claim is a materially different trust decision than a grounded one), tags, and the capture date. The review-gate mechanics (self-approval blocked, a different teammate's token succeeds) are unchanged — this is purely about giving the reviewer enough information to use that gate meaningfully rather than rubber-stamping a title.
Evidence: 2 tests in mcp/cloud-server.test.ts assert the full body text and the 'No cited paths' warning render for an ungrounded packet, and that a grounded packet's cited path renders under 'Cites:' with no false warning. Live-verified in the running dev server: a real repo packet's full multi-sentence body and its cited mcp/kernel.ts path both rendered correctly, screenshotted.
Verified by: npm test: 401/401 pass; live preview_screenshot against the running kage-cloud dev server
