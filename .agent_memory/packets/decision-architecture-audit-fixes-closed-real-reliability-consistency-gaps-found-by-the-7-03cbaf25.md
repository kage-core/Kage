# Architecture audit fixes: closed real reliability/consistency gaps found by the 7-area audit

> Followed up the kage architecture audit 2026 07 findings with real engineering fixes not just documentation , scoped …

Followed up the kage-architecture-audit-2026-07 findings with real engineering fixes (not just documentation), scoped to what could be verified and safely completed in one pass. Declined one item explicitly: writing genuine hook automation for the 13 non-claude-code, non-codex agents, since most of those tools have no equivalent hook mechanism to Claude Code's and faking automation that doesn't actually work would recreate the exact kind of unsubstantiated claim the prior two gimmick-removal passes existed to fix. The real fix for 'works for any agent' is hardening kage proxy (agent-agnostic by construction), not 13 separate fake integrations.

Fixes shipped: (1) mcp/kernel.ts setupAgent() -- kage setup <agent> --write silently wrote nothing to disk for 13 of 16 agents despite the flag implying it did; added a real write path for 10 JSON-config agents (cursor, windsurf, cline, roo-code, kilo-code, claude-desktop, openclaw, copilot, hermes, opencode) via a new upsertNestedMcpJson() merge helper, and gemini-cli via execFileSync of the gemini CLI itself with graceful failure handling. goose and generic-mcp deliberately stay print-only (goose's config.yaml is real YAML Kage has no safe writer for; generic-mcp has no known path). (2) claudeAmbientHookSummary()'s scriptPaths list was missing kage-edit-context.sh, so a missing/outdated edit-context hook was never flagged by doctor/verify -- added it. (3) verifyAgentActivation()'s ambient_hooks_present was hardcoded true for every non-claude-code agent (vacuously, since they have zero hook requirements), making --json output misleading; added a new ambient_hooks_supported field (true only for claude-code) so consumers can distinguish 'not applicable' from 'actually present'. (4) kage compact's CLI default (execute) was the opposite safety posture of kage_compact's MCP default (dry-run) for the identical compactProject() function; aligned CLI to dry-run-by-default, added --execute as the explicit opt-in. (5) kage_context -- the MCP tool CLAUDE.md calls the mandatory first step of every task -- had no CLI form, so nothing outside an agent session could reproduce it; extracted the assembly logic (previously duplicated inline in mcp/index.ts) into a shared kernel.ts kageContext() function and added a kage context CLI command, so both surfaces call the identical code. (6) Five more MCP-only tools had no CLI form (kage_search, kage_fetch, kage_list_domains, kage_learning_ledger, kage_workflow) -- extracted their logic into kernel.ts (kageListPublicDomains/kageSearchPublicGraph/kageFetchPublicGraphNode/KAGE_WORKFLOW_TEXT, reusing the existing kageSessionLearningLedger) and added kage community-domains/community-search/community-fetch/ledger/workflow CLI commands. (7) kage reverify had no MCP tool wrapper, so an agent that found a stale packet via kage_pr_check couldn't re-verify it in-session without shelling out; added kage_reverify (deliberately NOT promoted to CORE_TOOLS without eval evidence, matching the project's own eval-driven tool-surface discipline). (8) Added a proposal memory type (MEMORY_TYPES, OKF_STANDARD.md, okf.ts's TYPE_DISPLAY, the 3 high-value-type arrays, DECISION_INTELLIGENCE_TYPES, and a new inferLearningType heuristic checked BEFORE bug_fix so proposal language that mentions failure modes -- e.g. 'we should add retry logic because requests currently fail silently' -- doesn't lose to the bug_fix pattern) -- closes the taxonomy gap for capturing a feature idea/spec as a first-class, verification-compatible memory type, without building task tracking (assignment/priority/sprints), which correctly stays out of scope and belongs in GitHub Issues per the project's own existing restraint discipline. (9) gains/savings/metrics report three genuinely different things (real cumulative usage ledger; a deterministic reproducible benchmark; a whole-repo modeled estimate) with zero cross-reference, so a user asking 'how much has Kage saved me' got 3 unexplained numbers -- added one-line cross-references to each command's output rather than forcing them to compute identically, which would have been methodologically wrong.

Verified but NOT fixed, and explicitly not claimed as fixed: the proxy's live-subscription OAuth confirmation remains pending (unit-tested and code-verified correct -- system prompt untouched, memory appended to last user message -- but never run against a real live Claude Code subscription session, which this environment cannot safely do without live user credentials), and the hook/proxy double-capture risk when both are active for one session remains a structural, unsolved gap (the proxy has no visibility into Claude Code's real session id at the HTTP layer; the packet's own author already flagged this needs client-side cooperation Kage doesn't control). Both were re-verified as still accurately describing current, unmodified mcp/proxy.ts (11/11 proxy tests still pass, file untouched since commit e858ac3).
