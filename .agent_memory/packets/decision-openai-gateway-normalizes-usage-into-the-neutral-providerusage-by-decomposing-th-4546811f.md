# OpenAI gateway normalizes usage into the neutral ProviderUsage by decomposing the full prompt

> The OpenAI compatible gateway mcp/vnext/adapters/openai proxy.ts plus openaiGateway in gateway.ts maps OpenAI usage i…

The OpenAI-compatible gateway (mcp/vnext/adapters/openai-proxy.ts plus openaiGateway in gateway.ts) maps OpenAI usage into the SAME neutral ProviderUsage shape the Anthropic path uses, rather than adding a new usage type. Verified 2026-07-15: OpenAI reports the FULL prompt count including cached tokens in usage.prompt_tokens (chat/completions) or usage.input_tokens (responses), with the cached portion as a SUBSET under prompt_tokens_details.cached_tokens or input_tokens_details.cached_tokens. This is the opposite of Anthropic, whose usage.input_tokens is only the uncached remainder. So extractOpenAiUsage never sums: it stores the uncached remainder (full prompt minus cached) as the neutral input_tokens, the cached subset as cache_read_input_tokens, a measured zero for cache_creation_input_tokens (OpenAI bills no separate cache-write line), and completion/output tokens as output. The shared totalPromptTokens then returns exactly the whole prompt and the shared promptTokenBreakdown and buildTransformationReceipt work unchanged. That is what stops the cross-provider fake-saving bug: both receipt sides are the whole prompt, one commensurable quantity. Gemini (Task 3) has the same shape (usageMetadata.promptTokenCount full, cachedContentTokenCount a subset) and should reuse this exact normalization. Absent usage yields null (honestly unavailable), never zero.
Evidence: 28 new tests in mcp/vnext/adapters/openai-gateway.test.ts pass, including a cached response (prompt 10000, cached 9800) proving the receipt measures the whole prompt of 10000, never the remainder, so no fake saving. Full suite 698/698 plus dogfood 12/12.
Verified by: npm test --prefix mcp reports 698 pass, 0 fail; node --test over proxy.test.js, adapter.test.js, openai-gateway.test.js reports 102 pass
