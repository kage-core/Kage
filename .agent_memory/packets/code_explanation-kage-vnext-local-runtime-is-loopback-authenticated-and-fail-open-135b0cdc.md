# Kage vNext local runtime is loopback authenticated and fail-open

> The Phase A vNext runtime listens only on 127.0.0.1:3112 by default, exposes public GET /v2/health, and requires an e…

The Phase A vNext runtime listens only on 127.0.0.1:3112 by default, exposes public GET /v2/health, and requires an exact machine bearer token for status, handshake, event, context, and receipt routes. Its token and status files live under .agent_memory/daemon/vnext with atomic 0600 writes, untrusted filesystem object and symlink-ancestor rejection, a 2 MiB actual-byte JSON limit, and an idempotent close that terminates partial HTTP connections before closing SQLite and removes only its own unchanged live status file. Legacy startDaemon enables this boundary only through the internal vnext option, reports startup failures without stopping legacy routes, and closes the runtime on SIGTERM. The runtime imports the existing lazy SQLite boundary so legacy Node 18 module loading remains intact.
Evidence: TDD red failures covered missing runtime modules, symlinked runtime ancestors, and a partial request that blocked close. Final verification: build plus focused runtime/daemon tests passed 36/36; npm test --prefix mcp passed 495/495 and dogfood 12/12.
Verified by: npm run build --prefix mcp; cd mcp && node --test dist/vnext/runtime/server.test.js dist/daemon.test.js; npm test --prefix mcp
