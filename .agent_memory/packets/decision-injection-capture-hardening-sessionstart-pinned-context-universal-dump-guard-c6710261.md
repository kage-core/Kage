# Injection + capture hardening: SessionStart pinned context + universal dump guard

> Two fixes to the loop's weak halves. INJECTION: kageResume mcp/kernel.ts ~18164 — what the SessionStart hook injects …

Two fixes to the loop's weak halves. INJECTION: kageResume() (mcp/kernel.ts ~18164) — what the SessionStart hook injects — previously gave only the recent-timeline digest + usage policy, NOT the team's curated always-on memory. It now prepends renderPinnedRepoContext(readContextSlots(projectDir)) (the same pinned block recall's context_block uses at ~10479), so a new session starts holding the high-signal repo facts, not just instructions; has_content also flips true when a pinned block exists. CAPTURE: the serialized-dump guard in capture()/capturePersonal() (~16331, ~20236) is now universal — it fires on EVERY capture path (agent kage_learn AND the auto-distill/observation pipeline, which call capture() without strictCitations and were how 300KB dumps leaked in) and trips on isSerializedDumpTitle(input.title) OR isSerializedDumpBody(input.body), so junk is blocked at the source before it reaches even the pending inbox. Recall-side filtering remains the safety net. Verified: 352 unit + 12 dogfood tests pass; junk title and junk body both blocked at capture, clean learning captured; resume runs with pinned-context wiring. NOTE: the capture-guard title+body enhancement + its tests were largely already in the working tree; this packet records the combined state.
Evidence: npm test -> 352 pass + 12 dogfood pass; kage learn junk title -> 'Learning capture blocked', junk body -> 'Learning capture blocked', clean -> captured; kage resume runs and pinned wiring present at kernel.ts:18164 and :10479.
