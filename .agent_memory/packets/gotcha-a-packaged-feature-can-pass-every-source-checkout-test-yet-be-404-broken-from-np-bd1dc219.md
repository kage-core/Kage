# A packaged feature can pass every source-checkout test yet be 404-broken from npm — verify from an installed tarball

> 4.0.0 shipped the knowledge portal kage viewer /app/ broken for every npm user, while working perfectly from a source…

4.0.0 shipped the knowledge portal (kage viewer -> /app/) broken for every npm user, while working perfectly from a source checkout, so no test caught it. Two independent packaging faults: (1) the daemon resolved portal assets ONLY from the monorepo path resolve(__dirname,'..','..','platform','web','dist'), which exists in a checkout but never in an installed package; (2) package.json files:['dist/**/*.js'] silently excluded the portal's index.html and .css even if present, because the glob only matches .js. Root lesson: any feature whose assets live OUTSIDE the published package dir, or are non-.js, must be (a) copied INTO the package at build/publish time and (b) whitelisted in files[] by a non-.js glob, and MUST be verified by 'npm pack -> install into a clean prefix -> exercise the installed binary', not by running from the repo. Fix pattern shipped: resolvePortalDir() prefers bundled dist/app then falls back to monorepo; scripts/bundle-portal.mjs wired as postbuild (tolerant) + prepublishOnly (--require); files[] adds 'dist/app/**/*'. Verified by installing the REGISTRY 4.0.1 and curling /app/ -> 200.
Evidence: 4.0.0 global install: curl /app/ -> 404; npm pack --dry-run showed 0 platform/web files. 4.0.1 from registry: dist/app/index.html present, installed viewer serves /app/ -> HTTP 200, assets 200, SPA deep link 200.
Verified by: mcp/portal-serving.test.js 5/5; registry install of 4.0.1 serves /app/ 200
