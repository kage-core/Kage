# vNext hooks: a stale runtime must never silently disable Kage, and stand-down is per-event

> The vNext Claude adapter plugin/hooks/kage vnext adapter.sh posts evidence to the local runtime and lets the legacy h…

The vNext Claude adapter (plugin/hooks/kage-vnext-adapter.sh) posts evidence to the local runtime and lets the legacy hooks "stand down" when a runtime is live. Two failure modes are non-obvious and both silently destroy the memory harness, so they are now guarded:

1. FAIL-OPEN IS NOT ENOUGH — A STALE RUNTIME MUST NOT LOOK LIVE. status.json carries a pid and is removed only on graceful close, so SIGKILL/OOM/reboot leaves it behind. If the legacy hooks trust that file and stand down while the adapter's curl gets ECONNREFUSED and fails open, then ZERO paths run for every event, forever, silently — while kage doctor still reports healthy. Worse, if any other local process later binds that port, the adapter POSTs the user's raw prompt plus the bearer token to it and pipes the reply into the agent's systemMessage. So: a runtime counts as live only if the recorded pid is actually alive and ours, and status.json/token pass ownership+mode checks (0600 in a 0700 dir we own). Anything unverifiable means "no runtime", and the LEGACY path keeps running. The fail-safe direction is always: when in doubt, legacy runs.

2. STAND-DOWN MUST BE PER-EVENT, NOT GLOBAL. The adapter only handles 6 of the 9 Claude hook events. A blanket "runtime is live, so stand down" guard silently killed Stop, PreCompact, and SubagentStop — meaning kage refresh, kage pr summarize, the kage reconcile blocking gate, and distillation never ran again, and the vNext runtime does not produce packets in Phase A, so nothing replaced them. A legacy hook may stand down ONLY for events the adapter actually handles. The regression test walks every event in the shipped hooks.json and asserts exactly one effective handler per event — it fails on zero as well as on two.

Also: audit mode must NOT inject context. Audit exists to measure an uncontaminated baseline; if the hook injects there, the proxy's audit baseline already contains Kage context and the exact-vs-partial savings number comes out wrong (and in assist the context would be added twice). Only assist injects.

Also: file events must stay repo-scoped. The legacy read/edit hooks refuse paths outside the repo root; the adapter initially dropped that, so reading ~/.ssh/config or another employer's repo emitted a file_open event in the team_metadata (shareable) privacy class carrying that path. Content was always excluded; paths were not.
Evidence: Both failures were found by review of commit 7f5ca5c and fixed in 860a272. The per-event handler test walks every event in the shipped hooks.json against a live runtime and asserts exactly one effective handler — it fails on zero as well as on two.
Verified by: npm test --prefix mcp (586/586 at 860a272); node --test mcp/dist/vnext/adapters/adapter.test.js (39/39); dogfood 12/12
