# Dogfooding kage scan on Kage itself found 3 real bugs: ghost-export false positive, isRecord triplication, debt-marker self-match

> Ran kage scan project . on the Kage repo itself as a final cleanup verification triggered by a /goal directive to com…

Ran kage scan --project . on the Kage repo itself as a final cleanup verification (triggered by a /goal directive to complete the project with cleaned, structured, working code). Found and fixed 3 real issues, all confirmed by before/after re-scans and new regression tests.

(1) Ghost-export false positive: averageNumber in mcp/metrics-math.ts is genuinely imported and called 9+ times in mcp/kernel.ts, but truthReport's ghost-export detector flagged it as dead code. Root cause: the raw-text fallback check (which exists specifically to catch references the code graph misses) reads a file's full text only if size_bytes <= 512KB (mcp/kernel.ts:12473 area) -- but mcp/kernel.ts itself is ~900KB, so the fallback silently returned empty text for kernel.ts, meaning any symbol used ONLY from within kernel.ts (the single largest, most central file in the repo) could never be found by the safety net. Fixed by raising the cap to 5MB (this is a cheap, memoized single read, not the expensive AST-parsing MAX_CODE_FILE_BYTES cap used elsewhere) -- confirmed the false positive disappeared on rescan.

(2) isRecord triplicated: the exact same 2-line type guard was independently defined in mcp/kernel.ts (private), mcp/proxy.ts, and mcp/registry/index.ts. Kage's own duplicate-cluster detector caught the proxy.ts/registry pairing (kernel.ts's copy wasn't in the flagged pair, possibly because it's unexported). Consolidated into a new leaf module mcp/type-guards.ts (zero internal imports, following the same pattern already established by mcp/metrics-math.ts's own header comment: 'Leaf-first is the safe way to decompose the kernel'). Note: mcp/registry/index.ts cannot import from mcp/kernel.ts (kernel.ts already imports FROM registry/index.ts, so that direction would create a cycle) -- the leaf-module extraction was the only safe consolidation path, not a straight 'import from kernel.ts'.

(3) Debt-marker self-match: mcp/kernel.ts's own debt-marker detector implementation (a comment describing what it detects, and the DEBT_RE regex literal itself, which necessarily spells out TODO/FIXME/HACK/XXX/@deprecated/@todo) was matching its own detection regex, producing 8 fake 'unresolved debt markers' in kernel.ts and 3 in cli.ts once the size-cap fix above let the raw-text scan actually reach kernel.ts's full content for the first time. Fixed with a general heuristic, not a special-cased exclusion: when 2+ DIFFERENT marker keywords appear within 15 characters of each other, that's almost always a listing/description of the convention (e.g. 'TODO/FIXME/HACK', 'TODO, FIXME, or HACK'), not a real individual marker -- real, distinct comments are always separated by substantial prose (30+ chars, usually a different line). Verified this doesn't break the existing 2-marker positive-detection test (two genuinely separate '// TODO: ...' and '// FIXME: ...' comments on different lines, ~35 chars apart, still both count).
