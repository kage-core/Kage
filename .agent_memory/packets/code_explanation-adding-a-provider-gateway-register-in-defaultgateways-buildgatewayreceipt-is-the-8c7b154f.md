# Adding a provider gateway: register in defaultGateways; buildGatewayReceipt is the neutral receipt builder

> A second ProviderGateway OpenAI was added to the Kage proxy with no change to the neutral core in mcp/proxy.ts, confi…

A second ProviderGateway (OpenAI) was added to the Kage proxy with no change to the neutral core in mcp/proxy.ts, confirming the Task 1 seam holds. The pattern Task 3 (Gemini) should follow: put the provider wire functions in their own module (mcp/vnext/adapters/openai-proxy.ts mirrors anthropic-proxy.ts), define the gateway object in mcp/vnext/adapters/gateway.ts, and append it to defaultGateways (order matters only for overlapping paths; OpenAI and Anthropic paths are disjoint exact matches). One real Task 1 gap was fixed minimally: buildProxyReceipt in anthropic-proxy.ts hard-coded provider anthropic, so the receipt attribution logic (audit vs assist, forwarded vs original, before/after commensurability) was extracted into a provider-neutral buildGatewayReceipt(input, {provider, snapshots}); buildProxyReceipt now delegates with provider anthropic (byte-identical behavior, existing tests unchanged) and openaiGateway.buildReceipt delegates with provider openai plus OPENAI_PRICE_SNAPSHOTS. createTokenCounter returns null for OpenAI (no cheap count-tokens endpoint), so a transformed OpenAI receipt is honestly partial unless the response usage already reports the count. Price snapshots live in mcp/vnext/measurement/pricing.ts (OPENAI_PRICE_SNAPSHOTS): only high-confidence dated, sourced rates are encoded (gpt-4o, gpt-4o-mini, gpt-4.1 family); every other model resolves to null cost via the existing findPriceSnapshot path, which is the correct, expected outcome, not a bug.
Evidence: Full mcp suite 698/698 plus dogfood 12/12 after adding openaiGateway; mcp/proxy.ts unchanged (git diff shows only gateway.ts, anthropic-proxy.ts, pricing.ts modified plus two new adapter files).
Verified by: npm test --prefix mcp (698 pass, 0 fail); git diff --stat HEAD shows mcp/proxy.ts untouched
