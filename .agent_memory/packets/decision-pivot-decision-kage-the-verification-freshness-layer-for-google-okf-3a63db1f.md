# Pivot decision: Kage → the verification/freshness layer for Google OKF

> Decision 2026 06 28 : pivot Kage to become the TRUST/FRESHNESS layer for Google's Open Knowledge Format OKF, released…

Decision (2026-06-28): pivot Kage to become the TRUST/FRESHNESS layer for Google's Open Knowledge Format (OKF, released 2026-06-12). Rationale: OKF standardized the curated-knowledge store (a directory of markdown concept files with YAML frontmatter, each anchored to a `resource` URI, plus `# Citations`/`timestamp`/`log.md`) but EXPLICITLY left verification, freshness, staleness, and validation "out of scope for v0.1" — and OKF's "format, not platform" charter commits Google NOT to build the checker into the spec (verification needs live warehouse/repo credentials a format can't require). ~75% of Kage's existing engine (source_hash staleness, detectContradictions, truthReport/doc-lie scanner, verifyCitations, the deterministic multilingual code graph) IS that missing checker. New one-liner: "OKF tells your agent what's true. Kage proves it still is." Headline noun = freshness; cold-artifact negation = DRIFTED.

CRITICAL red-team-forced edit: treat OKF as ACCELERANT, NOT DEPENDENCY. The product verifies the rotting docs teams already have TODAY (dbt model descriptions, data-catalog columns, README-vs-code, OpenAPI-vs-live-endpoint) and emits/consumes OKF as the premier supported format — it must NOT require an OKF bundle to exist to do anything. This removes the fatal risk of betting the company on a 2-week-old format's adoption.

Product = one CLI + MCP, three verbs: `okf verify <target>` (hero + zero-install cold artifact; deterministic, NO LLM on the verdict path), `okf enrich` (producer; join paths from the graph not a hallucinating LLM pass), `okf serve` (MCP that withholds drifted concepts). Launch artifact: deterministic drift scan of Google's OWN public OKF sample bundles against their live BigQuery sources, signed + reproducible → Show HN.

Business model: open-core; bill on bundles + monitored resources (NOT seats); Free CLI → Solo $19 → Team $99 (hosted continuous monitoring + CI drift gate + dashboard) → Business $499 → Enterprise $25-120K/yr (VPC runner). ~90% margin (deterministic verdict path = cost moat). The MOAT lives in PLATFORM (continuous hosted monitoring, CI gate, tamper-evident drift ledger, multi-warehouse connectors, VPC runner) — the part OKF's charter commits Google not to build — NOT in the format/frontmatter convention. GTM discipline: publish drift RESULTS as bait, withhold detection METHOD; contribute only a thin spec-convention pointer, never the full implementation.

Codebase: substrate swap not rebuild — ~55-60% of 34.5K LOC survives (the engine: code graph, source_hash, detectContradictions, truthReport all KEEP/REFACTOR). RIP OUT the JSON packet store as user surface, ~55 of 75 MCP tools, ~50 of 70 CLI commands, the 15-agent install matrix. Verification metadata stored as OKF-legal custom `x-kage-*` frontmatter (anchors/verified-status/freshness/manifest) so bundles stay OKF-conformant. First slice: `okf verify` over the existing engine, ~1-2 weeks; first paying Team ~day 60. Full plan in OKF_PIVOT.md.

Top risk: Google folds verification into OKF v0.2. Mitigation: moat in platform not format; results-as-bait/method-withheld; be the neutral multi-cloud + code-and-cross-substrate reference verifier first.
Evidence: Multi-track planning workflow (4 design tracks: format-mapping, product, business/GTM, codebase-migration) + 4-attack red-team (Google-absorption, verification-triviality, OKF-adoption, buyer/budget) + xhigh synthesis. OKF spec verified from github.com/GoogleCloudPlatform/knowledge-catalog/okf/SPEC.md (verification/freshness explicitly out of scope; arbitrary producer frontmatter fields permitted).
