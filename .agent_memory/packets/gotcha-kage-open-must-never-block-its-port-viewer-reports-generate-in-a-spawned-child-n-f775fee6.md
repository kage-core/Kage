# kage open must never block its port: viewer reports generate in a spawned child, not the daemon event loop

> startViewer mcp/daemon.ts pre generated ~25 JSON reports SYNCHRONOUSLY before listen — on a loaded repo this one: 367…

startViewer (mcp/daemon.ts) pre-generated ~25 JSON reports SYNCHRONOUSLY before listen() — on a loaded repo (this one: 367 packets) that is MINUTES of kageRepoXray/capabilities/benchmark work with the port dead the whole time; kage open appeared broken (curl 000 for 90+ seconds). Two-step lesson: (1) moving the grind after listen() is NOT enough — synchronous work on the same event loop leaves the bound socket accepting connections it never answers (banners print, curls hang); (2) the fix is a SPAWNED CHILD PROCESS: generateViewerReports(projectDir) is exported from daemon.ts and mcp/viewer-reports.ts is the child entry; startViewer spawns it after listen() with stdio ignored. The viewer tolerates missing reports (404 -> honest empty states, filled on reload), so bind-first is strictly better UX. Verified: viewer + portal answer 200 within ~3s of launch; daemon tests 15/15. Debugging gotcha discovered on the way: daemon.test.js resolves viewer/ from process.cwd() — run it from mcp/ (as npm test does), or it fails with ENOENT and looks like a viewer regression when it is a wrong-cwd invocation.
Evidence: Before: curl 000 after 90s, process sampling showed event loop busy; after banner-print (post-listen sync attempt) curls hung on an accepting-but-unresponsive socket. After child-process fix: viewer 200 + portal 200 ~3s after launch; npm --prefix mcp node --test dist/daemon.test.js 15/15; full suite 1474/1474.
Verified by: curl probes before/after; node --test dist/daemon.test.js from mcp/; npm test --prefix mcp
