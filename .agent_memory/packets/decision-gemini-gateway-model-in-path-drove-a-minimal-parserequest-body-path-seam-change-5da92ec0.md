# Gemini gateway: model-in-path drove a minimal parseRequest(body, path?) seam change

> The Gemini ProviderGateway Task 3 is the third gateway on the neutral proxy core, added by putting all Gemini wire se…

The Gemini ProviderGateway (Task 3) is the third gateway on the neutral proxy core, added by putting all Gemini wire semantics in mcp/vnext/adapters/gemini-proxy.ts and wiring geminiGateway in mcp/vnext/adapters/gateway.ts (registered in defaultGateways as [anthropic, openai, gemini]). Gemini differs structurally from the other two: the MODEL is in the request PATH (/v1beta/models/{model}:generateContent), not the body. This forced the one seam change in the whole workstream: ParsedProviderRequest.parseRequest gained an optional second arg, parseRequest(body, path?). Anthropic/OpenAI keep reading model from the body and ignore the arg (TypeScript structural typing makes their existing 1-arg impls valid). Only mcp/proxy.ts's model-extraction call site passes clientReq.url; composeInjection stays path-free because it only needs lastUserText, which is always body-derived. Eligibility matches a POST whose path (query stripped) is resource:action with action in {generateContent, streamGenerateContent} and resource ending /models/{model} — disjoint from /v1/messages and /v1/chat/completions|/v1/responses (those have no resource:action colon). The ?key=... and x-goog-api-key auth ride in query/headers which the core forwards untouched. Usage decomposition mirrors OpenAI exactly: usageMetadata.promptTokenCount is the FULL prompt incl. cached (total = promptTokenCount directly, NOT a sum), cachedContentTokenCount is the cached subset and is ABSENT (not 0) on a cache miss (treat falsy as 0, never crash), candidatesTokenCount is output, cache_creation = 0. Streaming usage/functionCall parse from BOTH a JSON array of chunks (no alt=sse) and SSE (?alt=sse), usageMetadata only on the final chunk. functionCall has no id, so tool_result evidence disambiguates by block_index. createTokenCounter returns null (honest partial): Gemini's :countTokens is real but model-specific, and the model is in the path which the counter seam {upstream, headers} does not carry — extending a second seam to exact-measure a deprioritized path would be gold-plating. Verified against https://ai.google.dev/api/generate-content and https://ai.google.dev/api/tokens on 2026-07-15.
Evidence: Build clean (tsc), targeted node --test 130/130, full suite 726 pass / 0 fail, dogfood 12/12. Gemini file alone: 26/26.
