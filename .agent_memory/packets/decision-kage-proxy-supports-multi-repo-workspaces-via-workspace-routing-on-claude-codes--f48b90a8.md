# kage proxy supports multi-repo workspaces via --workspace, routing on Claude Code's real Primary working directory

> Extended the proxy from one repo per process to serving an entire workspace of repos from one process. New pure resol…

Extended the proxy from one-repo-per-process to serving an entire workspace of repos from one process. New pure resolveRequestProjectDir(defaultProjectDir, workspaceRoot, body) in proxy.ts parses 'Primary working directory: <path>' out of the request's system prompt (confirmed against a REAL live captured Claude Code 2.1.202 request, not guessed — system is a multi-block array with an Environment section containing that exact line). Safety boundary: multi-repo routing is entirely OFF (always the fixed --project) unless --workspace is passed; even then, a candidate path is only honored if it resolves to workspaceRoot itself or a real filesystem descendant of it, else falls back to the default project. This is READ-ONLY on system (parses it, never modifies it), preserving the existing OAuth-safety invariant (system must stay byte-identical or subscription auth 429s). Each request's hasMemory/injectMemory/observe now resolve against the per-request project dir instead of a value computed once at startup.
Evidence: New e2e test 'one proxy process serves two different repos under --workspace, each getting its own memory' spins up a real startProxy() server and proves repo A's request only ever sees repo A's memory, repo B's only repo B's. 4 unit tests for resolveRequestProjectDir (default-pinned when no workspace, in-workspace routing, out-of-workspace rejection, missing-path/no-system-text fallback). 396/396 full suite pass.
Verified by: npm test: 396/396 pass
