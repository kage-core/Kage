# Injection-relevance eval harness is the acceptance gate for corpus-normalized relevance

> benchmarks/injection relevance kage.mjs is the repeatable eval for composeInjection's ungated injection decision reca…

benchmarks/injection-relevance-kage.mjs is the repeatable eval for composeInjection's ungated injection decision (recall projectDir/query/4/false, no relevance gate). It imports the REAL compiled composeInjection from mcp/dist/proxy.js through a recording fake gateway — never a mirrored copy — and cross-checks a byte-identical diagnostic recall (trackAccess: false, run BEFORE the production call) against the production decision; any disagreement is a per-case error, so the eval cannot silently drift from production. It seeds a 3-packet and a 150-packet store at runtime via kernel capture() (deterministic in-file corpus with deliberate collision carriers: single, word, reply, pong, good, looks, continue, fix, hello, thanks) and scores content-free / real-relevant / real-but-absent labeled queries. Recorded 2026-07-16 baseline (exit 0, measured not aspired): injection_precision 0.1538, false_injection_rate large 1.0 (every content-free prompt injects — the pong problem), small 0.3333 ("ok continue" and "fix it" inject even on 3 packets), small_store_recall 1.0, expected_top_hit_rate 1.0, absent_injection_rate 0.875. The score bands reproduce the floor inversion in-harness: a relevant small-store paraphrase hit scores 25.03, inside the large store's content-free noise band (max 34.64), and "fix it" (34.39) outscores a real question's top hit (30.80) — confirming the negative result that an absolute injection score floor is impossible. This harness is the acceptance gate for the corpus-normalized relevance kernel task: a real fix moves false_injection_rate and absent_injection_rate down and injection_precision up WITHOUT moving small_store_recall or expected_top_hit_rate down. Run node benchmarks/injection-relevance-kage.mjs (--json for per-case table, --assert-baseline to gate regressions, exit 1 only on regression or case errors) or npm run bench:injection --prefix mcp. It is NOT part of npm test. Gotcha: initProject auto-creates a repo_map "repo structure" packet, so seeded stores hold seeded+1 approved packets; the harness keys unseeded hits as auto:<type> so the per-case table stays deterministic despite the random temp-dir token in that packet's title.
Evidence: Two consecutive runs produced byte-identical metrics, score bands, verdicts, and per-hit scores (31 cases, 0 errors); --assert-baseline exits 0 on the recorded baseline and exits 1 when a metric is forced to regress; full suite stayed 746/746 + dogfood 12/12 with the harness excluded from npm test.
Verified by: node benchmarks/injection-relevance-kage.mjs --assert-baseline (exit 0, deterministic across runs); npm test --prefix mcp (746 pass, 12 dogfood pass); commit 2051253
