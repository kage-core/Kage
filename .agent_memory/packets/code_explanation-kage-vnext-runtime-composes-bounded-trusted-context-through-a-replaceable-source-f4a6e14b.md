# Kage vNext runtime composes bounded trusted context through a replaceable source

> The Phase A vNext local runtime preserves its loopback only, bearer authenticated, lease hardened lifecycle and now c…

The Phase A vNext local runtime preserves its loopback-only, bearer-authenticated, lease-hardened lifecycle and now composes automatic context through a provider-neutral ContextSource. The HTTP boundary strictly projects repository/task identities, dense target arrays, query, changed files, and a positive bounded token budget before consulting the source. The capsule builder ranks whole trusted sections deterministically, estimates tokens as ceil(UTF-8 bytes / 4), never truncates bodies or citation identifiers, returns honest empty capsules, and derives stable content IDs with a five-minute expiry. The only vNext module importing the legacy kernel is context/legacy-source.ts, which adapts approved repo-local recall, verification contracts, and target risks while explicitly excluding stale, disputed, negative-feedback, personal, and team memory. Runtime source failures or explicit unavailability return a non-leaking 503, while an empty trusted result remains a 200 empty capsule; composition does not persist delivery receipts.
Evidence: Strict TDD first failed on the four missing context modules. The new context suite passed 13/13, runtime suite passed 37/37, npm test --prefix mcp passed 522/522, and dogfood passed 12/12.
Verified by: npm run build --prefix mcp; node --test mcp/dist/vnext/context/context.test.js mcp/dist/vnext/runtime/server.test.js; npm test --prefix mcp
