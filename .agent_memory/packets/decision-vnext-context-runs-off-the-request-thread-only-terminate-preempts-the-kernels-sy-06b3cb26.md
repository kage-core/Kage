# vNext context runs off the request thread; only terminate() preempts the kernel's synchronous work

> Kage's vNext /v2/context composes context on a persistent worker thread WorkerContextSource + context worker.ts , not…

Kage's vNext /v2/context composes context on a persistent worker thread (WorkerContextSource + context-worker.ts), not on the runtime's request thread (commits 8daa017, 6b25d23).

The reason, which should not be re-litigated: recall, kageRisk, and kageTeammateBrief are SYNCHRONOUS kernel calls, and kageRisk falls back to a full buildCodeGraph on a cold repo. On the runtime's single event loop they block /v2/health, /v2/events, and /v2/receipts for the entire analysis, so Task 5's fail-open adapters time out and evidence events are LOST.

The key insight: a timeout on the request thread would be a lie, because the timer cannot fire while the loop is blocked. Only worker.terminate() can preempt synchronous CPU work, and that is what makes the deadline real rather than decorative. Never "fix" a blocking synchronous call with a Promise.race timeout — it returns early while the loop stays blocked.

Design constraints that follow, each with its reason:
- Persistent worker, not per-request: the 22k-line kernel module and the code-graph cache are both expensive to load, and per-request workers would pay both on every call.
- A terminate() is always followed by a respawn, because a persistent worker would otherwise stay poisoned by one runaway job.
- The 60s worker deadline is a RUNAWAY KILLER, not a latency budget. Adapters abort at ~500ms client-side and fail open, so they never wait on it. A legitimate cold code-graph build takes tens of seconds and must be allowed to finish, because killing it means it never completes.
- After a deadline kill the source drops the queued jobs and fails fast for a cooldown (default 30s). Reason: buildCodeGraph writes graph.json ONLY on completion, so a repo whose cold build exceeds the deadline never persists a graph — the respawned worker restarts the same build, gets killed again, and /v2/context 503s forever while each request pins a core for a full timeout. An out-of-band `kage refresh` warms graph.json and is the intended recovery; any success clears the cooldown.
- LocalRuntimeHandle exposes the ContextSource the runtime resolved, purely so a test can assert the shipped default is off-thread. Without that assertion, reverting the default to an on-thread source is a silent one-line regression that every other test still passes — a reviewer demonstrated exactly that.

LegacyContextSource is unchanged and now runs inside the worker. It is still the only vNext module importing mcp/kernel.ts.
Evidence: Measured against the old on-thread path: while a 2s context request was in flight the runtime answered only 1 health probe inside a 600ms window; with the worker it answers continuously and still accepts /v2/events. The default-wiring guard was mutation-checked by reverting the compiled server.js default to LegacyContextSource, which fails the test.
Verified by: npm run build --prefix mcp; node --test mcp/dist/vnext/context/context.test.js mcp/dist/vnext/runtime/server.test.js (71/71); npm test --prefix mcp (547/547); dogfood 12/12
