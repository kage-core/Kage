# Recall scores are not corpus-normalized: an absolute injection floor is impossible (measured)

> MEASURED NEGATIVE RESULT — do not re attempt an absolute recall score floor for injection. The extraction quality gap…

MEASURED NEGATIVE RESULT — do not re-attempt an absolute recall-score floor for injection.

The extraction-quality gap is real: the proxy's composeInjection (mcp/proxy.ts) and the capsule source (legacy-source.ts) inject whatever recall returns with NO relevance gate. On a 325-packet store, a content-free prompt like "Reply with the single word: pong" recalls 4 lexical accidents at scores 23-33 (matching common words like "single"/"word" in packet titles) and all 4 get injected — kilobytes of irrelevant memory attached to a prompt that asked for nothing. Real questions score 40-75 on the same store, so a floor of ~35 looks obvious.

IT IS NOT. Recall scores are match-strength sums, NOT corpus-normalized relevance: on a small store, a direct, genuinely-relevant single-packet match scores just 14.3 — below the big store's junk band (20-33). Any absolute floor that removes big-store noise also silences small/new repos entirely, which is the onboarding case the proxy exists for. The kernel already returns zero hits for pure-stopword queries on small stores, so the noise problem only manifests at scale.

The right fix is in kernel scoring, not a bolt-on: recall needs a corpus-normalized relevance signal (or per-hit match-quality reasons exposed to callers) so injection sites can distinguish "weak because irrelevant" from "weak because the store is small". Until then, injection remains ungated by deliberate choice — noise on large stores was judged less harmful than killing injection on small ones. The per-request receipts/deliveries now being collected in the live audit are exactly the data to tune this properly later.

Also fixed while auditing the loop (2026-07-16): three approved junk packets (two ungrounded "Resume:" session artifacts + one raw "Tool failed:" command dump predating the dump-capture fix) marked stale via kage feedback in BOTH stores and verified withheld from recall (suppressed count 3); two pending auto-distill junk drafts (a raw shell command line and a file-changed observation) rejected. The auto-distill quarantine itself worked correctly — junk landed in pending, excluded from recall.
Evidence: Content-free prompts on the 325-packet store: "Reply with the single word: pong" -> 4 hits at 33.0/29.4/24.1/23.4 (all lexical accidents, all INJECTED into a live assist session); "thanks, looks good" -> 24.8; "ok continue" -> 29.3; "fix it" -> 43.9. Real questions: oauth 74.9, run-tests 71.8, workspace 62.1, receipts-pricing 40.5. THE KILLER: a seeded 1-packet store with a direct-match query ("how do payment retries dedupe on the ledger" against the payments-idempotency packet) scores only 14.3 — BELOW the big store's junk band. A MIN_RECALL_INJECTION_SCORE=35 gate was implemented, then reverted before commit when calibration showed it would kill all injection for small/new repos.
Verified by: Live measurement 2026-07-16 against the real 325-packet master store and a seeded 1-packet store, via the compiled kernel recall exactly as composeInjection calls it
