# Kage vNext local runtime uses hardened leases and a SQLite singleton lock

> The Phase A vNext local runtime remains loopback only and bearer authenticated, but its filesystem and ownership boun…

The Phase A vNext local runtime remains loopback-only and bearer-authenticated, but its filesystem and ownership boundary is now explicitly lease-based. Startup secures the current-user-owned runtime directory to 0700 and revalidates its device/inode ownership lease around token, lock, main database migration, server bind, and status publication. A persistent 0600 runtime-lock.db is opened through the hardened lazy SQLite boundary and holds BEGIN IMMEDIATE for the runtime lifetime; competing runtimes receive the existing already-running error, release rolls back/closes without unlinking, and process death releases the OS lock. Exact repeated handshakes are accepted without replacing started_at, while conflicting authoritative task identity fields return 409. JSON uses fatal UTF-8 decoding, token/status writers clean owned temporary files on failure, and startup/normal close independently attempt server, main database, status, and lock cleanup while preserving the primary failure. Legacy daemon startup remains optional and fail-open, and imports remain safe on Node without node:sqlite.
Evidence: Strict TDD reproduced permissive existing directories, foreign ownership, directory replacement after database open, conflicting handshake identity, duplicate runtimes, invalid UTF-8, and cleanup failures. Focused runtime/daemon verification passed 47/47. npm test --prefix mcp passed 506/506 and dogfood 12/12, including the CLI import-safety test without node:sqlite.
Verified by: npm run build && node --test dist/vnext/runtime/server.test.js dist/daemon.test.js (from mcp); npm test --prefix mcp
