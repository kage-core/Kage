# kage check is the hero verb: drift verification, counted not estimated

> Kage's product pivot executed 2026 07 03: kage check mcp/check.ts verifies claims in agent context files CLAUDE.md, A…

Kage's product pivot executed 2026-07-03: `kage check` (mcp/check.ts) verifies claims in agent-context files (CLAUDE.md, AGENTS.md, .cursor/rules, README, docs/**) against the code — cited paths, npm scripts, make targets, CLI subcommands — with three honest buckets: confirmed drift / verified true / unverifiable. It is deliberately independent of the code graph and memory store so it runs cold on any repo in seconds and leaves nothing behind. Corpus-validated on 70 public repos with agent-context files: 21 repos (30%) had confirmed drift, 91 findings, 1033 verified, ~85% sampled precision; four tuning passes cut raw findings from 339 to 91 by adding 10 precision rules (fence-aware, build dirs, placeholders, env-var roots, gitignored=runtime, package-relative suffix match, aspirational-doc exclusion, hedged lines, top-segment-must-exist, single-segment skip). Full methodology in docs/CHECK_VALIDATION.md. In the same change the fabricated-metrics surface was removed: bytes/4 gains receipts no longer ship in kage_context/kage_recall output (counts of stale-withheld only), the "Show the Value" relay section was deleted from CLAUDE.md, and the "18% faster than grep" N=3 claim and self-graded trust badge were removed from README. Rationale: the audit showed the estimate-based receipt destroyed exactly the trust a verification product needs; every user-facing number must now be a count of reproducible checks. Teams adopt via `kage check --init-ci` (PR comment + gate on new drift only, baseline command for legacy adoption).
Evidence: 70-repo corpus scan results in scratchpad corpus-results/; hand-adjudicated 35-finding and 30-finding samples; npm test 376/376 + 12/12 dogfood passing
Verified by: npm test (376 pass, 0 fail); kage check --project . returns 0 confirmed / 8 verified on this repo; corpus run: 70 repos, 0 errors
