# GOTCHA: kage proxy tagged every observation session_id literally 'default', risking collision with real hook sessions

> proxy.ts's observe call never set session id, so kernel.ts's observationHash defaulted every proxy captured observati…

proxy.ts's observe() call never set session_id, so kernel.ts's observationHash defaulted every proxy-captured observation to the literal string session:'default'. Since observationHash is scoped by repo+type+session+...+minute-bucket, every kage proxy run ever, across restarts, on the same repo, shared ONE dedup bucket, and diverged from whatever real Claude Code session id a UserPromptSubmit hook records the SAME conversation under (a real UUID) meaning the identical prompt text could be captured twice (once via proxy tap, once via hook) with no way for hash-based dedup to catch it, since the session component differs. FIX: startProxy() now generates one stable per-process session id (proxy-<uuid> via node:crypto randomUUID) at server start and passes it on every observe() call. This does not eliminate the hook/proxy dual-capture risk when both are active for one session (the proxy has no visibility into Claude Code's real session id at the HTTP layer, and that would need client-side cooperation) but it does fix within-run dedup and makes proxy-captured observations distinguishable in .agent_memory/observations instead of colliding under a shared 'default' bucket across unrelated runs.
Evidence: New e2e test spins up a real startProxy() server against a fake upstream over an actual HTTP round-trip, then reads back the written observation via the newly-exported loadObservations() and asserts session_id matches /^proxy-/ and is not 'default'.
Verified by: npm test: 391/391 pass (proxy.test.js: 6/6)
