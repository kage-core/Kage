# Live A/B: memory's measured value is CONDITIONAL — big when knowledge is not in the code, ~zero when it is

> The reuse savings thesis is now VALIDATED with a real agent, and the honest answer is that memory's value is CONDITIO…

The reuse-savings thesis is now VALIDATED with a real agent, and the honest answer is that memory's value is CONDITIONAL on whether the knowledge is already in the current code:

- When the answer IS derivable from the code the agent will read anyway (Scenario A): injected memory adds NO measurable value. N=5 showed cost -1.2% (noise), identical turns (4/4 every trial), identical correctness (5/5). Claude Code verifies by reading source regardless of injected context, and control finds the fact itself. The injected memory is redundant. (The N=1 pilot's "56% cheaper" was pure prompt-cache-accounting noise that the N=5 mean erased — always run N>1.)

- When the answer is NOT in the current code — a cross-session fact, an incident history, a decision rationale, a why (Scenario B): memory's value is large and measured. Correctness went 0/3 -> 3/3 (control literally cannot answer and says so), AND it was cheaper (-41%, $0.070 -> $0.041) and faster (turns 8.33 -> 3.33), because the control agent wastes turns hunting for knowledge that isn't in the repo before giving up. Here memory's primary win is CORRECTNESS/CAPABILITY (answering something otherwise impossible), with token/cost savings as a secondary consequence of not thrashing.

Implication for the product's savings claim: the flat estimated "~30M tokens saved" that `kage gains` reports is not what the real numbers show in EITHER scenario (neither ~0% nor a fixed rate) — the value is conditional and lives specifically in out-of-code knowledge. Kage should position memory around cross-session/non-derivable knowledge (gotchas, incidents, decisions, why), NOT around re-deriving facts already visible in the code, and should not headline a single token-savings number.

Method that worked (reusable): real `claude -p --output-format json --setting-sources project` (drops global hooks, keeps OAuth) with ANTHROPIC_BASE_URL at an assist proxy; measure input+cache_creation+cache_read (total prompt), num_turns, total_cost_usd; VERIFY injection per-arm (proxy injected-count and/or a HTTP-level forwarded-body check) so a proxy-startup race can't silently invalidate a trial — an earlier un-instrumented batch produced a false 0/3-treatment result exactly because of that race. The end-to-end A/B is the only thing that settles this; the deterministic bench:reuse answer-from-memory coverage is a proxy, not the token delta.
Evidence: Two scenarios, real agent runs measuring claude's own reported usage + num_turns + total_cost_usd. Scenario A (fact IS in the code, N=5): control(no memory) vs treatment(memory) were statistically identical — cost -1.2% (within per-trial noise), turns 4.0 vs 4.0 every trial, correctness 5/5 vs 5/5; the agent read the source in both arms and found the fact regardless. Scenario B (fact NOT derivable from code — a cross-session incident record, N=3, injection verified): control 0/3 correct (answered "I don't see information about PAY-4471 in this repository"), 8.33 turns, $0.070; treatment 3/3 correct (answered "truncating to 24 characters... full 52-character base32 digest" — content only in the injected packet), 3.33 turns, $0.041. Injection independently proven: recall returns the packet (score 25.3) and the proxy's forwarded body contains the "Verified repo memory ## PAY-4471" block; an instrumented run showed "7 memories injected" and a correct answer.
Verified by: Live A/B 2026-07-17: real headless Claude Code (claude -p --output-format json, haiku, subscription OAuth) through kage proxy --mode assist, global hooks dropped via --setting-sources project so the proxy was the only memory channel; injection confirmed at the HTTP layer and via the proxy's injected-count.
