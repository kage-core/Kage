# Default MCP surface is 3 vNext verbs (context/retrieve/feedback); the 12-tool core is now KAGE_TOOLS=legacy

> Phase E Task 10 commit 76e4819 cut the default product surface over to vNext. MCP listTools now returns EXACTLY three…

Phase E Task 10 (commit 76e4819) cut the default product surface over to vNext. MCP listTools() now returns EXACTLY three verbs by default, in canonical order: kage_context, kage_retrieve, kage_feedback. The pre-vNext 12-tool core (CORE_TOOLS in mcp/index.ts, the old eval-driven core=12 with kage_check as hero verb) is NO LONGER the default — it is the body of KAGE_TOOLS=legacy, which exposes the full legacy registry with a Deprecated note + migration-doc link on every non-survivor tool. KAGE_TOOLS=full remains the complete internal registry; KAGE_TOOLS=vnext is an explicit spelling of the default. CLI: primary `kage help` leads with connect/status/open/doctor/export/migrate; `kage install`/`up`/`run`/`down`/`context`/`check`/`setup` remain primary Getting started commands (the background-proxy lifecycle is unchanged). Every deprecated invocation prints one supported replacement + a v5 removal notice + docs link to stderr and records ONLY the command name + version locally (never arguments); `kage legacy <cmd>` runs a deprecated command deliberately; `kage legacy scan` finds scripts/config still invoking legacy commands. Map lives in mcp/vnext/migration/legacy-command-map.ts; docs at docs/migration/v4-command-map.md + v4-upgrade.md. Local-first fail-open and the frozen wire protocol are untouched.
Evidence: git show 76e4819 (cutover commit, backend 1374/1374 + deploy 36/36 + dogfood 12/12 green); mcp/index.ts:116 CORE_TOOLS comment now reads "The pre-vNext core (12 tools). It is NO LONGER the default surface — it is the body of KAGE_TOOLS=legacy"; live `node mcp/dist/cli.js help` output shows the v4 surface with up/run/down still primary.
Verified by: git show 76e4819; node mcp/dist/cli.js help; npm test --prefix mcp (per commit message: 1374/1374)
