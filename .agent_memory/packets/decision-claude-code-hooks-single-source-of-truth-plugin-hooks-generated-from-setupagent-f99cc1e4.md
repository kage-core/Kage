# Claude Code hooks: single source of truth — plugin/hooks generated from setupAgent

> The Claude Code plugin's hooks are GENERATED, not hand maintained. generatePluginHooks mcp/kernel.ts runs setupAgent …

The Claude Code plugin's hooks are GENERATED, not hand-maintained. generatePluginHooks (mcp/kernel.ts) runs setupAgent("claude-code") into a sandbox HOME and derives plugin/hooks/*.sh + plugin/hooks/hooks.json from its output, re-targeting hook command paths from `~/.claude/kage/hooks/` (the npm install path) to `${CLAUDE_PLUGIN_ROOT}/hooks/` (the plugin path). The hook script bodies are path-agnostic (they shell out to `kage`), so the same bodies serve both channels. Exposed as the maintainer CLI `kage gen-plugin-hooks [--plugin-dir <dir>]` (mcp/cli.ts). setupAgent is the SINGLE SOURCE; the committed plugin/hooks are generated artifacts, guarded by the unit test "plugin hooks are generated from setupAgent — committed plugin/hooks must not drift" (mcp/kernel.test.ts) which asserts byte-equality and fails CI on drift. TO CHANGE A CLAUDE CODE HOOK: edit the setupAgent script templates in kernel.ts, run `kage gen-plugin-hooks`, and commit the regenerated plugin/hooks. Before v2.5.5 the two had drifted — the plugin shipped no observe.sh, so plugin users got recall + file-context but none of the observe/auto-distill capture pipeline; now both have full 9-event parity. Removed the plugin-only kage-prompt-context.sh (subsumed by observe.sh).
Verified by: 355 unit + 12 dogfood tests pass (incl the drift-guard test); bash -n clean on all 5 generated scripts; hooks.json valid JSON (9 events); generated == committed verified independently; published v2.5.5 + global upgraded
