# SDLC work-item pipeline (Phase 1): WorkStage state machine, Kage-as-substrate boundary, non-forgeable cloud gate

> Kage pivoted this session from a passive memory layer toward an agent driven SDLC platform "agents drive, humans appr…

Kage pivoted this session from a passive memory layer toward an agent-driven SDLC platform ("agents drive, humans approve gates"), at explicit user request after growth/discovery pushes on repo memory alone produced no real usage. Phase 1 shipped a WorkStage state machine (proposed -> claimed -> in_review -> done) as new optional fields (stage, claimed_by, claimed_at) on MemoryPacket, scoped to type: "proposal" packets only. transitionWorkStage() in mcp/kernel.ts is the single writer for .stage (mirroring, and deliberately avoiding, MemoryStatus's known fragility of 42+ scattered writers) and validates the WORK_STAGE_TRANSITIONS table (proposed->claimed, claimed->in_review|proposed, in_review->done|claimed, done->[]). claimWorkItem() and linkImplements() are thin wrappers; linkImplements() reuses the existing upsertPacketEdge() for implements/implemented_by edges (the same pattern supersedeMemory() already used) and auto-advances the proposal claimed->in_review — no new packet type needed for the output, any existing type works. A claim-race guard (withWorkItemLock(), an exclusive "wx" lock file per packet id) guards the check-then-write in transitionWorkStage.

The single most consequential decision, made explicitly: Kage orchestrates work-item STATE, never COMPUTE. It defines/exposes the state machine (CLI: kage claim/implements/stage/gate; MCP: kage_claim_work_item/kage_link_implements/kage_list_work_items/kage_transition_work_item; daemon REST: GET/POST /kage/work-items...) that external agent runners poll, claim, and advance themselves. Kage's own daemon/cloud server never holds an API key and never spawns an agent process — that boundary is written into docs/CLOUD.md's new "SDLC work items" section so it doesn't erode under future feature pressure.

The in_review -> done transition (the only terminal one) is deliberately unreachable via any scriptable, agent-callable path: kage_transition_work_item (MCP tool) rejects to_stage: "done" outright, and kage stage --to done refuses non-interactively. It's reachable only two ways, with explicitly different trust levels: (1) kage gate review, a local TTY-interactive reviewer prompt whose self-block compares plain git-config-derived actor strings (spoofable, documented as weak); (2) kage cloud approve, which is the actually non-forgeable gate — it reuses cloud-server.ts's pre-existing, unmodified bearer-token-hash self-approval block (a submitting token can never approve its own packet), and after a genuinely different teammate's token approves server-side, new glue in cli.ts's "cloud approve" handler applies transitionWorkStage(..., "done") locally (checking the local packet is type: proposal at stage: in_review before doing so). cloud-server.ts itself needed zero code changes — packet_json already stores the whole packet so stage/claimed_by ride through push/approve for free, verified by a new cloud-server.test.ts test ("a work item's stage and claimed_by ride through push/approve unmodified").

Also fixed as a Phase 1 prerequisite: daemon.ts's readPacketTitle() did a raw JSON.parse() on packet files, which silently failed on the now-primary .md OKF format and fell back to the raw filename as a fake title in the SSE live-feed payload — replaced with format-aware readPacketFromDisk() via a new readPacketSummary() that also carries packet_type/stage/claimed_by.

Verified end-to-end twice: the full automated suite (429 main + 12 dogfood tests, 0 failures) and a live manual CLI smoke test through the real cloud path (propose -> claim -> implement -> kage implements auto-push -> a second teammate's kage cloud approve -> confirmed local stage: done applied, visible in the packet's embedded kage-state JSON).
Evidence: Full test suite green; live smoke test confirmed proposal packet's stage field reached "done" only after a genuinely different teammate's kage cloud approve, never via any MCP tool or non-interactive CLI flag
Verified by: npm test: 429/429 + 12/12 pass; live manual CLI smoke test through the real cloud approval path
