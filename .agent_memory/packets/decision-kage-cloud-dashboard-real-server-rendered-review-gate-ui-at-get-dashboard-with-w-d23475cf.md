# Kage Cloud dashboard: real, server-rendered review-gate UI at GET /dashboard, with working Approve/Reject wired to the actual API

> Added GET /dashboard?team=<id &token=<token to mcp/cloud server.ts: a server rendered HTML page renderDashboard , zer…

Added GET /dashboard?team=<id>&token=<token> to mcp/cloud-server.ts: a server-rendered HTML page (renderDashboard(), zero build step, zero new dependencies) listing a team's pending and approved packets, with real Approve/Reject buttons that call the SAME POST /v1/teams/:id/packets/:packetId/approve|reject endpoints the CLI uses via client-side fetch() — the page never bypasses the API's review-gate logic, it's a second real client of it, not a shortcut. Token travels in the query string (same trust boundary as passing it on the CLI; the server remains explicitly self-hosted-only, no TLS/rate-limiting). Built because the prior 'Kage Cloud v1.5' work shipped API-only with no way to SEE the review gate happen; the dashboard was requested specifically so a human could watch self-approval get blocked and a real second-teammate approval succeed, live, by clicking. Verified end-to-end in the actual running dev instance, not just tests: pushed 212 real repo packets to a demo team, clicked Approve as the submitting owner token -> blocked in the browser with the real 403 self_approval_blocked message rendered inline; switched to an invited teammate's token, clicked Approve on the same class of item -> succeeded, page reloaded, pending/approved counts moved for real (207->206, 5->6) against the live SQLite DB.
Evidence: 4 new/passing tests in mcp/cloud-server.test.ts: dashboard renders correct pending/approved counts and packet titles, a query-string-less request shows the team/token entry form, an invalid token 401s. Full suite 400/400 pass. Live browser verification via Claude Preview tools: real click on Approve as the packet's own submitter produced the rendered error 'a submitter cannot approve their own packet — have a teammate review it'; a second click as a different invited token succeeded and the DOM-rendered counts changed from the real API response.
Verified by: npm test: 400/400 pass; live preview_click against the running kage-cloud dev server, DOM counts confirmed before/after

## Verification
