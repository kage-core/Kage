# Portal Playwright specs must goto /app-prefixed paths — the SPA is mounted under /app/, not the origin

> The Phase C knowledge portal platform/web is a History API routed React SPA that the Kage daemon serves ONLY under /a…

The Phase C knowledge portal (platform/web) is a History-API-routed React SPA that the Kage daemon serves ONLY under /app/ (mcp/daemon.ts resolveAppAsset: /app and /app/ -> index.html, unknown /app/* -> index.html SPA fallback; bare /app redirects to /app/). platform/web/src/router.ts portalBase() detects the mount by checking pathname startsWith('/app'), and its own comment warns that "a full-page navigation to /review (rather than /app/review) would leave the mount." Therefore Playwright e2e specs MUST navigate with /app-prefixed absolute paths (page.goto('/app/'), '/app/review', '/app/tasks/:id', '/app/runbooks/:slug', '/app/integrations'). A leading-slash goto like page.goto('/review') resolves against the ORIGIN (new URL('/review', baseURL) discards baseURL's path), lands off the SPA mount, and fails even against a correctly seeded CI daemon. The original Task-9 specs had this bug; fixed post-gate in commit 8dcde5e. Also: the browser journeys need a LIVE daemon serving the built portal under /app/ backed by a seeded repo model (features, runbook slug rotate-signing-keys, a review item, task-1); there is no in-repo seed harness — CI supplies KAGE_PORTAL_URL and playwright.config.ts skips honestly (never green-washes) when it is absent.
Evidence: Verified: new URL('/app/review','http://127.0.0.1:3113/app') -> http://127.0.0.1:3113/app/review (correct) vs new URL('/review',...) -> http://127.0.0.1:3113/review (off-mount). npx playwright test --list shows 12 tests in 6 files compile after the fix. mcp/daemon.ts resolveAppAsset SPA-falls-back unknown /app/* to index.html.
Verified by: node URL resolution check; npx playwright test --list; git show 8dcde5e
