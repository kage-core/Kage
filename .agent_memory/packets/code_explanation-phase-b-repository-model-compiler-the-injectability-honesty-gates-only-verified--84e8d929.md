# Phase B repository-model + compiler: the injectability honesty gates (only verified/approved, shadow proposes, no auto-verify)

> Kage vNext Phase B added an evidence backed repository model mcp/vnext/repo model/ + knowledge compiler mcp/vnext/com…

Kage vNext Phase B added an evidence-backed repository model (mcp/vnext/repo-model/) + knowledge compiler (mcp/vnext/compiler/) on the Phase A SQLite/protocol foundation. The load-bearing invariant, enforced at every layer, is INJECTABILITY TRUST: only claims with trust_state "verified" or "approved" may reach an agent; "proposed"/"disputed"/"stale"/"superseded"/"archived" never inject (isInjectableTrustState in repo-model/types.ts). TrustState is a new type added additively to protocol/types.ts but is explicitly NOT a wire field — no frozen protocol-v1 message changed.

Three honesty gates the review loop had to enforce (each was a real bug caught + hardened):
1. A successful agent command auto-verifies a runbook claim ONLY when grounded on the repository's EXACT declared bare script invocation (from package.json scripts + the repo's own package manager derived from its lockfile). It must reject: shell metacharacters (so `npm test && rm -rf /` never becomes a verified injectable runbook), trailing args/flags (`npm test --coverage`), a mismatched runner (`pnpm test` in an npm repo), and npx. Otherwise a non-declared, possibly destructive command would be grounded on evidence that doesn't actually support it. (extractors/command.ts, harden ee14225.)
2. Model-assisted extraction (compiler/model-extractor.ts) runs in SHADOW mode behind an injected provider-neutral seam (no vendor branch, no network in-module): it redacts secrets before anything leaves, validates evidence ids/entity kinds, floors every candidate to "proposed" (never verified/injected), and fails open to zero candidates on provider error — leaving deterministic compilation unchanged. (harden c973fda: NaN clamp, URL/snippet redaction.)
3. OKF export never mints "verified" from a human approval alone — a human "approved" claim exports as approved, not verified; only an evidence check earns "verified". (Task 9 harden 1215240.)

The model-backed ContextSource (context/model-source.ts) injects only verified/approved claims and is enabled only after shadow comparison passes. Storage migration is at v5. This whole phase is the substrate Phase C (portal), Phase D (context efficiency), and Phase E (team) build on.
Evidence: Phase B completed 2026-07-18 as 10 tasks (commits 307653a..0405107), each through an implement -> adversarial-review -> harden loop; 6 hardening commits fixed real honesty-boundary bugs the reviews caught.
Verified by: npm test --prefix mcp 938/938 + dogfood 12/12; Phase B gate mcp/vnext/phase-b-gate.test.ts 2/2; build clean; migration v5
