# kage cloud link + kage viewer Team sidebar link: local viewer surfaces the hosted dashboard without duplicating it

> Answered 'why can't the cloud dashboard live inside kage viewer' by linking the two instead of merging them. Architec…

Answered 'why can't the cloud dashboard live inside kage viewer' by linking the two instead of merging them. Architecturally they stay separate on purpose: kage viewer is repo-local, single-user, unauthenticated, reads .agent_memory directly off disk for ONE checkout; kage cloud's dashboard is cross-repo, multi-user, authenticated, and mutates a separate server's SQLite DB via a review-gated API — folding one into the other would force viewer to grow an auth/multi-tenant dimension it doesn't need for its actual job. Instead: new 'kage cloud link --project <dir> --server <url> --team <id> --token <token>' persists the association to .agent_memory/team/link.json (writeTeamLink/readTeamLink in kernel.ts; gitignored, same trust tier as the token already living in CLI args and the dashboard URL). startViewer (mcp/daemon.ts) reads this at boot and writes it into a new pre-generated report (viewerReportPaths().teamLink -> .agent_memory/reports/team-link.json, included automatically in viewerUrl()'s query params since that function iterates all report keys). The frontend (mcp/viewer/console.js) fetches it independently of the main render pipeline and, if present, un-hides a 'Team ↗' link in the sidebar's external-links section (mcp/viewer/index.html) pointing straight at the cloud server's own /dashboard URL — clicking it navigates to a genuinely different server, so it belongs with Website/Docs/GitHub, not the internal SPA nav buttons. No link configured -> the link stays hidden and nothing else about viewer changes.
Evidence: 3 new tests: readTeamLink/writeTeamLink round-trip + overwrite-in-place in kernel.test.ts, teamLink present in viewerReportPaths + viewerUrl's query string in daemon.test.ts. 402/402 full suite pass. Linked this actual repo to its real demo cloud team via the CLI and confirmed .agent_memory/team/link.json is correctly untracked by git.
