# Pages CI failing: cloud-server.ts eagerly imported node:sqlite, crashing every kage command on Node <22.5

> The user reported 'the website CI is failing'. Investigated via gh run list/gh run view: the Pages workflow docs site…

The user reported 'the website CI is failing'. Investigated via gh run list/gh run view: the Pages workflow (docs site deploy) had failed on every push for at least the last 5 commits, always in ~23-27s. The actual error: Error [ERR_UNKNOWN_BUILTIN_MODULE]: No such built-in module: node:sqlite, thrown from mcp/dist/cloud-server.js:19:23 during node mcp/dist/cli.js refresh.

Root cause: mcp/cloud-server.ts (the Kage Cloud self-hosted team server, built earlier this session) had a top-level, unguarded ES import -- import { DatabaseSync } from "node:sqlite" -- but node:sqlite is a Node 22.5+ builtin, while mcp/package.json declares engines: {node: ">=18"}. Because ES/CJS module imports are evaluated eagerly at load time, and mcp/cli.ts unconditionally imports startCloudServer from ./cloud-server.js at its own top level, EVERY kage CLI command (not just cloud ones) crashed immediately on any Node version below 22.5 -- before main() or any command dispatch even ran. The MCP server (mcp/index.ts) was unaffected since it never imports cloud-server.ts.

Discovered why this was invisible in the main CI workflow: every other GitHub Actions workflow (ci.yml, kage-pr.yml, kage-sync.yml, release.yml, kage-benchmark.yml) pins node-version: 22, which resolves to a patch version well past 22.5 and has node:sqlite available -- masking the bug. Only pages.yml pins node-version: "20" (the actual declared minimum-ish), which is why only Pages caught it. Decided to KEEP Pages pinned to Node 20 rather than bumping it to match the others -- it is now valuable, deliberate coverage of the declared engines>=18 floor and would have caught this bug immediately when it was introduced, instead of silently shipping a broken CLI to any real Node 18-22.4 user for multiple releases.

Fix: converted the import to type-only (import type { DatabaseSync } from "node:sqlite", erased at compile time) and moved the actual runtime require("node:sqlite") to be lazy, inside openCloudDb() -- the single function that actually constructs a DatabaseSync -- guarded by try/catch, matching the exact pattern mcp/kernel.ts's queryClaudeMemStore() already used for the same reason. If node:sqlite is unavailable, openCloudDb() now throws a clear, actionable error ("kage cloud requires Node 22.5+... every other kage command works fine on Node 18+") instead of crashing the whole CLI. Verified with a real subprocess regression test (mcp/cloud-server.test.ts) that shadows node:sqlite via Module._load to throw the exact Node-20 error, then confirms cli.js still loads and runs -- confirmed this test genuinely fails against the old eager-import code (reproduced the exact ERR_UNKNOWN_BUILTIN_MODULE stack trace from the real CI log) before the fix, and passes after.
