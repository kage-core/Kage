# Recall ranking: stopword dilution + ungated priors buried exact matches (fixed v3.2.0)

> Two stacked holes made ambient recall hooks pass raw prompts as queries bury exact matches. 1 STOPWORDS was tiny 18 w…

Two stacked holes made ambient recall (hooks pass raw prompts as queries) bury exact matches. 1) STOPWORDS was tiny (18 words), so filler words like what/can/we earned BM25+vector credit, and because knowledge-graph edge facts embed packet titles, a title containing 'what' collected up to 18 graph points across its edges for a query sharing only stopwords. 2) In recallBreakdown, quality (up to ~10) and freshness (2) were unconditional priors while usage was already relevance-gated — so an entrenched high-quality packet scored ~12-15 with ZERO query match and crowded real matches out of top-5. Fix: expanded STOPWORDS with interrogatives/pronouns/auxiliaries, and gated quality/freshness on matchSignal (coreRelevance + pathTypeTag), mirroring the existing usage gate. Diagnosis trick: recall --explain --json exposes score_breakdown; the tell was a #1 result with bm25:4 (one stopword) but graph:18 + quality:10. Reference-type packets are extra vulnerable (graphWeight 0, pathTypeTag x0.2). Also added observation retention: refreshProject prunes .agent_memory/observations older than KAGE_OBSERVATION_RETENTION_DAYS (default 30, 0 disables) — the dir had grown 12k files/48MB with no cap.
Evidence: Before: recall 'what is gbrain? can we replace it' ranked 5 packets matching only stopwords, gbrain packet absent; after: gbrain packet #1 on recall, prompt-context, and kage_context. Regression tests: 'natural-language filler words do not outrank a rare exact term' + 'refresh prunes observations past the retention window'. 374 tests green.
Verified by: npm test (374 pass); live A/B on repo queries pre/post fix
