# kage up / kage run: the one-command proxy onramp and its safety split

> kage up and kage run mcp/cli.ts dispatch, logic in mcp/vnext/runtime/commands.ts are the one command onramp for the p…

`kage up` and `kage run` (mcp/cli.ts dispatch, logic in mcp/vnext/runtime/commands.ts) are the one-command onramp for the proxy-primary path. Design decisions future agents must preserve: (1) upProject() is a spawn-free PLAN — it runs connectProject (byte-idempotent audit-only config; --mode can never leak into the config on disk) and probes the port, but the CLI starts the foreground proxy AFTER printing instructions; that seam is why tests never launch long-lived processes. (2) `up` defaults to --mode audit (safe onboarding default) while bare `kage proxy` keeps assist for back-compat — this asymmetry is deliberate and documented in both help texts; do not "fix" it. (3) `up` is idempotent: a listener already on the port => same instructions, exit 0, never a failure; the runtime singleton lease surfaces as "already running ... reusing it". (4) `kage run -- <cmd>` sets ANTHROPIC_BASE_URL (and nothing else) in the child env, inherits stdio, propagates the child exit code, ignores SIGINT while the child runs (Ctrl-C belongs to the agent), and NEVER auto-starts the proxy — up owns that lifecycle; run fails fast (bounded 400ms TCP probe via probeLocalPort, host 127.0.0.1) with a one-line `kage up` hint. (5) startProxy's banner is now mode-aware: audit mode may not claim "will inject".
Evidence: 9 new tests in mcp/vnext/runtime/commands.test.ts (up config/plan, mode split, honesty strings, live-listener idempotency, fail-open runtime, run fail-fast/env/exit-code, help text); full suite 755/755 + dogfood 12/12; live smoke: kage up on a scratch project started runtime+proxy, second up exited 0 reusing both, kage run printed the injected ANTHROPIC_BASE_URL and propagated exit 7.
Verified by: npm test --prefix mcp (755 pass, 0 fail) + manual end-to-end smoke on ports 8991/8993
