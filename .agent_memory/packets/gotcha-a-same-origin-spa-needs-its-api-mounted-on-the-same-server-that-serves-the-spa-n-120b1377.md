# A same-origin SPA needs its API mounted on the SAME server that serves the SPA, not just somewhere

> The knowledge portal SPA platform/web fetches /v2/... SAME ORIGIN main.tsx: new KageApi '', token — empty base = rela…

The knowledge portal SPA (platform/web) fetches /v2/... SAME-ORIGIN (main.tsx: new KageApi('', token) — empty base = relative). The /v2 read API (mcp/vnext/api/router.ts: matchPortalRoute/handlePortalRoute) was implemented and tested, but only the vNext RUNTIME server (mcp/vnext/runtime/server.ts) mounted it. The kage viewer daemon (mcp/daemon.ts startViewer) served the SPA static bundle under /app/ but ZERO /v2 routes — so every panel showed 'Repository knowledge is unavailable: Kage API 404' even though the API code existed. Lesson: when a SPA fetches its API same-origin, the server that serves the SPA MUST also mount the API; a route existing in the codebase or on a different server/port does not help a same-origin fetch. Fix: servePortalApi() in daemon.ts mounts the read routes by reusing openRepositoryModel() (the read-safe open kage export/migrate already use — opened+closed per request, never takes the runtime writer lock, safe under SQLite WAL alongside a live runtime) + the pure handlePortalRoute() dispatcher (honesty contract preserved). Verified: 7 /v2 routes 200, undefined /v2 404, browser /app/overview renders real data, no console errors. Two layers here: 4.0.1 fixed the static SHELL (bundling), 4.0.2 fixed the DATA API (this).
Evidence: Live: curl /v2/overview 404 before, 200 after; browser Overview rendered recalls served 160 / stale withheld 514 / 229 packets with no console errors. Tests mcp/portal-serving.test.js servePortalApi 200/404 cases.
Verified by: mcp/portal-serving.test.js 8/8; full suite 1427/0; browser render verified
