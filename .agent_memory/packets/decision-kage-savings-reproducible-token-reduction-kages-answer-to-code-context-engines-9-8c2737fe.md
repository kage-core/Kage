# kage savings: reproducible token-reduction % (Kage's answer to code-context-engine's 94%)

> Added benchmarkSavings mcp/kernel.ts + the 'kage savings' CLI command as Kage's legible, screenshot able token saving…

Added benchmarkSavings (mcp/kernel.ts) + the 'kage savings' CLI command as Kage's legible, screenshot-able token-savings number, in response to elara-labs/code-context-engine (CCE) hitting 288 stars in 10 weeks on the description 'Save 94% on AI coding tokens'. Mechanism: reuse the existing deterministic benchmarkTaskComparison (baseline = full-file reads of the files a query touches; with-Kage = recall + code-graph context) across queries auto-derived from the repo's own code graph (deriveSavingsQueries: most-called exported functions/classes by call-edge count + declared routes, deterministic ordering), aggregate to a headline context_reduction_percent. On this repo: 97% (66K->2K tokens/query, recall hit 100%). Two honest advantages over CCE: (1) NO LLM on the measurement path, so same commit => identical percent (test asserts determinism) — CCE uses embeddings whose recall drifts; (2) queries auto-derived not hand-picked, removing cherry-pick risk. Carries CCE's own honest caveat verbatim: measured vs full-file reads, NOT vs the agent's real grep/partial-read behavior, so real-world savings run lower. STRATEGIC NOTE: this makes Kage competitive on CCE's axis but does not win — CCE has a 10-week head start + a conference talk (the distribution EVENT). The number arms any positioning (including the Frisk security pivot) but the breakout still needs an event, not just the metric.
Evidence: kage savings on this repo prints 97%; two runs identical (reproducibility verified); 383 unit + 12 dogfood tests pass incl. the new determinism test; CCE facts from gh api repos/elara-labs/code-context-engine (288 stars, desc 'Save 94%') + its README benchmark methodology
Verified by: npm test (383+12 pass); node dist/cli.js savings --project . run twice, identical 97%
