# Ambient hooks never silently die: PATH → baked cli.js → npx package-runner fallback

> The 1 activation bug is fixed: all Claude Code hooks generated by setupAgent session start, observe, stop, read/edit …

The #1 activation bug is fixed: all Claude Code hooks generated by setupAgent (session-start, observe, stop, read/edit-context) used `command -v kage || exit 0`, which silently no-oped for every npx-installed user because npx puts no `kage` binary on PATH — the entire ambient memory loop was dead on the documented install path. The fix (mcp/kernel.ts, claude-code branch of setupAgent) bakes a `hookKageResolve` block into every hook: try PATH first (fast), then `node <install-time cli.js path>` guarded by a runtime -f check (so npx cache pruning degrades gracefully), then `npx -y --package=@kage-core/kage-graph-mcp kage` as the never-dead fallback. NOTE: plain `npx -y @kage-core/kage-graph-mcp` runs the WRONG bin (the MCP server, which hangs on stdin) — the --package= form selecting the `kage` bin is required. Plugin hooks are regenerated from the same templates via `kage gen-plugin-hooks` (the drift test in kernel.test.ts fails if you forget). Verified end-to-end: sandbox with kage absent from PATH, observe.sh exit 0 AND index files actually written to the project.
Evidence: Sandbox test: env -i PATH without kage, observe.sh wrote .agent_memory/indexes/*.json; kernel.test.ts asserts package-runner fallback present in all hooks
Verified by: npm test 376/376 + 12/12; sandbox proof run 2026-07-03
