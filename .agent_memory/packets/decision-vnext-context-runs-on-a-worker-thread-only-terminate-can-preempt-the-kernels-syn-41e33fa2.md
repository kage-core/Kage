# vNext context runs on a worker thread; only terminate() can preempt the kernel's synchronous work

> Legacy memory imported from Markdown.

Evidence: Measured against the old on-thread path: while a 2s context request was in flight the runtime answered only 1 health probe in a 600ms window; with the worker it answers continuously (>=5 probes) and still accepts /v2/events. The default-wiring guard was mutation-checked by reverting server.ts to LegacyContextSource in dist, which fails the test.</evidence>
<parameter name="learning">Kage's vNext /v2/context composes context on a persistent worker thread (WorkerContextSource + context-worker.ts), not on the runtime's request thread. The reason is specific and worth not re-litigating: recall, kageRisk, and kageTeammateBrief are SYNCHRONOUS kernel calls, and kageRisk falls back to a full buildCodeGraph on a cold repo. Run on the runtime's single event loop they block /v2/health, /v2/events, and /v2/receipts for the whole analysis, so Task 5's fail-open adapters time out and evidence events are LOST.

A timeout on the request thread would be a lie: the timer cannot fire while the loop is blocked. This is the key insight — only worker.terminate() can preempt synchronous CPU work, which is what makes the deadline real rather than decorative. Do not "fix" a blocking sync call with a Promise.race timeout; it returns early while the loop stays blocked.

Design constraints that follow, each with a reason:
- Persistent worker, not per-request: the kernel module (22k lines) and the code-graph cache are both expensive to load, and per-request workers pay both every call.
- terminate() is always followed by a respawn, since a persistent worker would otherwise stay poisoned by one runaway job.
- The 60s worker deadline is a RUNAWAY KILLER, not a latency budget. Adapters abort at ~500ms client-side and fail open; they never wait for this. A legitimate cold code-graph build takes tens of seconds and must be allowed to finish, because killing it means it never completes.
- After a deadline kill, queued jobs are dropped and the source fails fast for a cooldown (default 30s). Reason: buildCodeGraph writes graph.json ONLY on completion, so a repo whose cold build exceeds the deadline never persists a graph — the respawned worker restarts the same build, is killed again, and /v2/context 503s forever while every request pins a core. An out-of-band `kage refresh` warms graph.json and is the intended recovery; any success clears the cooldown.
- The runtime exposes the ContextSource it resolved (LocalRuntimeHandle.contextSource) purely so a test can assert the shipped default is off-thread. Without that assertion, reverting the default to an on-thread source is a silent one-line regression that every other test still passes — a reviewer demonstrated exactly that.

LegacyContextSource is unchanged and now runs inside the worker; it remains the only vNext module importing mcp/kernel.ts.
Verified by: npm run build --prefix mcp; node --test mcp/dist/vnext/context/context.test.js mcp/dist/vnext/runtime/server.test.js (71/71); npm test --prefix mcp (546/546); dogfood 12/12
