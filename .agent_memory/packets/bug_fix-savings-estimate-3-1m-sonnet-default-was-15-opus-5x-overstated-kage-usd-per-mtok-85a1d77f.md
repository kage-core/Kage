# Savings $ estimate: $3/1M Sonnet default (was $15 Opus, ~5x overstated), KAGE_USD_PER_MTOK override

> VALUE DOLLARS PER MILLION TOKENS mcp/kernel.ts was hardcoded 15 $/1M input with a comment mislabeling it "Sonnet clas…

VALUE_DOLLARS_PER_MILLION_TOKENS (mcp/kernel.ts) was hardcoded 15 ($/1M input) with a comment mislabeling it "Sonnet-class" — but $15/1M is Claude Opus input pricing, so every "$ saved" receipt (kage gains, the MCP gains line, the site receipt) overstated dollar savings ~5x for the typical user (Sonnet ~$3/1M, Haiku ~$0.80, GPT-4o ~$2.50). Fixed in v2.5.7: the constant now defaults to a conservative $3/1M (Sonnet-class) and reads a float override from process.env.KAGE_USD_PER_MTOK (e.g. 15 for Opus). It is exported and consumed by recallDollarsSaved (kernel.ts:~3220) and the kage gains footer (mcp/cli.ts), which now prints the rate it used + the override hint ("via KAGE_USD_PER_MTOK" vs "Sonnet-class default") so the assumption is never hidden. Site receipt corrected $6.18→$1.24 (412K tokens at $3/1M, labeled configurable). The unit test now references the constant instead of a hardcoded 15. Lesson: never value tokens-saved at the most-expensive model tier by default; it inflates a number users already distrust as vanity.
Verified by: 355 unit + 12 dogfood pass; kage gains prints '$3/1M (Sonnet-class default)' by default and '$15/1M (via KAGE_USD_PER_MTOK)' when overridden; site receipt shows $1.24; published v2.5.7
